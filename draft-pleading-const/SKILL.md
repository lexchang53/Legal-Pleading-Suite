## 核心定位與工作流
本技能為純粹的**收斂與排版引擎**。假設使用者已經透過先前的對話或研究，產出了憲法違憲論點的大綱。

**步驟一：整理與輸出 Markdown 草稿**
請將使用者的論點素材，嚴格套入以下的 Markdown 骨架。

**步驟二：請求確認**
請將產出的 Markdown 草稿展示給使用者，並詢問是否同意。

**步驟三：編譯 Word 與 ODT 檔**
在使用者同意後，請使用本技能內建的 `scripts/build_const_pleading.py` 包裝腳本完成編譯與後處理：
```bash
python "<draft-pleading-const>/scripts/build_const_pleading.py" "草稿的絕對路徑"
```
其中 `<draft-pleading-const>` 是本技能資料夾的實際路徑。不要在 `SKILL.md` 中寫入特定使用者的絕對路徑；包裝腳本會依照自己的位置，自動尋找同層的 `draft-pleading` 與 `docx-to-odt` 技能。

可選參數：
```bash
python "<draft-pleading-const>/scripts/build_const_pleading.py" "草稿的絕對路徑" --docx "輸出.docx" --odt "輸出.odt"
```

（請注意：包裝腳本內部會依序執行三個不可省略的步驟：第一步為底層排版引擎，第二步為憲法書狀專屬 DOCX 後處理，第三步為 DOCX→ODT 轉檔與 ODT 後處理。不要再另外執行 `fix_odt_outline.py`，避免二次覆寫 `docx-to-odt` 已完成的大綱與清單修復。）

**ODT 排版穩定性規則**
- 區塊標題（如 `聲請審查客體`、`應受判決事項之聲明`、`主要爭點`、`聲請理由`、`壹、...`、`貳、...`）由 `postprocess_const_pleading.py` 套用 `書狀_區塊標題` 樣式處理。
- 不要用刪除 DOCX XML 節點（例如移除 `w:ind`）的方式修正區塊標題縮排；這會破壞 LibreOffice 判斷多層清單與編號重設的依據。
- ODT 的通用層級大綱、懸尾標點與行號 outside，交由 `docx-to-odt/scripts/convert_docx_to_odt.py` 統一處理。

## 憲法聲請書 Markdown 骨架標準

請嚴格遵守以下格式，特別注意：
1. **當事人欄位**：「姓名/名稱」與「身分證明文件字號：」必須在同一行，地址必須換行。
2. **標題字眼**：請精準使用 `聲請審查客體`、`應受判決事項之聲明`、`主要爭點`、`聲請理由` 等字眼。
3. **引用檢查**：若提及「系爭規定」、「系爭判決」、「確定終局裁判」，或引用「釋字」、「憲判字」、「德國聯邦憲法法院裁判」等段落時，必須以 `「...」` 引號精準包覆原文，否則排版將會報錯中斷！
4. **附件清單**：清單標題為 `附件：`，下方項目使用 `附件1：`、`附件2：`。

### 骨架範本：

```markdown
# 裁判及法規範憲法審查聲請書

案號：　　年度　　字第　　號（若案件尚未分案，則省略）

聲請人 [姓名]　身分證明文件字號：[字號]
　　住：[地址]
　　電話：[電話]

代表人 [姓名]　身分證明文件字號：[字號]
　　住：[地址]

訴訟代理人 [姓名]律師
　　事務所：[地址]
　　電話：[電話]

茲依憲法訴訟法第59條之規定，聲請裁判及法規範憲法審查事：

**　　聲請審查客體**
（內容...）

**　　應受判決事項之聲明**
（內容...）

**　　主要爭點**
（內容...）

**　　聲請理由**
**壹、本件聲請符合受理要件**
一、原因案件事實及所經過之訴訟程序
(一) ...

**貳、應受審查客體之違憲理由**
一、確定終局裁判所適用之法規範牴觸憲法
(一) ... 依系爭規定「（原文）」之意旨...

附件：
附件1：用盡審級救濟之最終裁判及其送達日期之證明
附件2：（應予廢棄之）不利確定終局裁判

謹狀
憲法法庭　公鑒
中華民國115年5月10日

具狀人：[姓名]
代表人：[姓名]
訴訟代理人：[姓名]律師
```
