# twlegalrag-search

本技能透過 `twlegalrag` CLI 檢索台灣法院裁判、產生可引用的判決封包，並檢查後續回答或書狀中的引用是否正確。

當任務需要建立在真實、可追溯、可驗證的台灣法院裁判資料上，而不是憑空摘要或未驗證的法律敘述時，應使用本技能。

## 開始前先檢查

先確認執行環境中可使用 `twlegalrag` 指令。

如果系統尚未安裝 `twlegalrag`，或無法執行該指令，先明確提醒使用者必須先安裝：

```bash
pip install twlegalrag
```

在本技能應使用 `twlegalrag` 的情況下，不要自行改用 HTTP 請求、網頁爬取或其他替代檢索方式來繞過這個依賴。

## 使用時機

當使用者有下列需求時，使用本技能：

- 查找特定法律爭議在台灣法院的裁判、裁判字號、理由摘要或相關片段。
- 在起訴狀、答辯狀、準備書狀、法律備忘錄或法律意見書撰寫前，蒐集可引用的實務見解。
- 將檢索到的裁判整理成可供後續引用的 bundle。
- 檢查 AI 生成的法律分析、書狀或摘要中，判決引用是否與封包內容一致。
- 解析 `citation_id`，例如 `[J1]`、`[J2]`，並對應到底層裁判資料。

## 指令規範

請直接使用 Python 執行 `twlegalrag` CLI，不使用 PowerShell 包裝腳本。

常用指令如下：

```bash
twlegalrag health
twlegalrag search "<查詢詞>" -n <數量> --read
twlegalrag pack "<查詢詞>" -o <輸出路徑.json> -n <數量>
```

如果檢索結果之後要被引用、驗證或交由下游生成法律文本，優先使用 `pack`。

**引用檢查統一使用 `scripts/check.py`**，不要直接呼叫 `twlegalrag check`：

```bash
python scripts/check.py <bundle.json> <answer.txt>
```

腳本會先驗證兩個檔案是否存在，再呼叫底層 `twlegalrag check` 並回傳退出碼，供流程判斷是否通過。若回答或書狀中已有 `[J1]`、`[J2]` 等標記，在交付前必須以此腳本驗證。

## 法院優先順序規則

在檢索、整理與引用裁判時，依下列順序優先處理：

1. 最高法院裁判優先。
2. 其次為臺灣高等法院及其分院裁判。
3. 若前述法院裁判不足以支撐檢索目的，再補充其他法院裁判。

當多筆裁判都與爭點相關時，應優先整理與引用較高審級法院的裁判，並在摘要與輸出中依法院層級排序，而不僅依原始檢索順位排列。

如果 bundle 同時包含不同法院層級的裁判，應先檢視最高法院裁判是否已足以支撐主要論證；只有在不足時，才補充高等法院及其分院裁判作為輔助。

## 查詢筆數規則

除非使用者另有指定，預設查詢筆數如下：

- `twlegalrag search` 預設使用 `-n 5`，用於快速確認檢索方向與結果相關性。
- `twlegalrag pack` 預設使用 `-n 6`，優先取得較精簡、較容易整理的核心裁判集合。
- 若法律爭點較廣、結果分散、或最高法院裁判不足，才逐步提高到 `-n 8` 或 `-n 10`。
- 若法律爭點明確且集中，或僅需驗證少量核心裁判，可降到 `-n 3` 至 `5`。
- 除非有明確理由，不要一次檢索過多筆結果，以免降低相關性並增加後續引用檢查與摘要負擔。

## 查詢詞建構原則

理解使用者的法律爭點後，依下列原則建構查詢詞，再進入標準流程。

### 建構公式

以「**法律構成要件關鍵字 + 法律效果關鍵字**」為基礎組合查詢詞：

- 競業禁止（構成要件）+ 違約金過高（效果）→ `競業禁止 違約金 過高`
- 租賃（構成要件）+ 終止（效果）→ `租賃契約 終止 拒絕返還`
- 董事忠實義務（構成要件）+ 損害賠償（效果）→ `董事 忠實義務 損害賠償`

### 查詢詞擴展規則

擴展查詢詞時，**只允許書寫變體擴展**，嚴禁跨越法律效果邊界：

| 擴展類型 | 是否允許 | 說明與例子 |
|---------|---------|-----------|
| **書寫變體** | ✅ 允許 | 同一概念的不同文字表達：條款 ↔ 約款、返還 ↔ 回復、損害 ↔ 損失 |
| **上下位概念縮小** | ⚠️ 謹慎 | 縮小至具體子概念尚可，但不得反向擴大：侵權行為損害賠償 → 侵權行為 ✅ |
| **跨法律效果擴展** | 🔴 **禁止** | 法律效果不同的相鄰概念互換：終止 → 解除、無效 → 撤銷 |

**台灣法律語境中絕對不可互換的概念對：**

- 🔴 **終止 ≠ 解除**：終止向將來生效（ex nunc），適用繼續性契約（租賃、勞動）；解除溯及既往（ex tunc），適用一次性契約（買賣），有回復原狀義務。
- 🔴 **解除 ≠ 撤銷**：解除消滅契約效力；撤銷針對意思表示瑕疵（詐欺、錯誤），法律依據與效果均不同。
- 🔴 **無效 ≠ 撤銷**：無效自始不生效力，無須主張；撤銷須當事人以意思表示行使，且有除斥期間。
- 🔴 **合意解除 ≠ 法定解除**：前者須雙方合意，後者源於法定或約定解除權，爭點方向完全不同。

### 多爭點處理

若案件同時涉及多個法律概念（如同時主張解除契約與損害賠償），以**各自獨立的查詢詞分別檢索**，再合併整理結果，不得將不同概念混合在單一查詢詞中以免稀釋相關性。

### 正反雙向查證原則（避免確認偏誤）

在進行法律實務見解檢索時，絕對不可僅憑單一方向的「支持性」關鍵字進行檢索而落入確認偏誤（Confirmation Bias）。為提供全面且具防禦性的訴訟風險評估，必須執行正反雙向查證：
1. **正面檢索**：尋找支持該主張的案例（例如：「撤銷定暫時狀態處分 情事變更」）。
2. **反面檢索（絕對必要）**：主動加入反向關鍵字，尋找法院「駁回」、「敗訴」或「不予准許」的反面案例與理由（例如：「撤銷定暫時狀態處分 駁回 實體爭執」）。
3. **綜合評估**：在交付摘要或生成法律意見時，必須同時並陳正反兩派實務見解，並主動揭示可能導致敗訴的「實務陷阱」或法官不予實體審查的邊界條件。

## 標準流程

依下列順序執行：

1. 先以 `twlegalrag health` 確認檢索服務可用。
2. 理解使用者的法律爭點，整理成適合檢索的自然語言查詢詞。
3. 先使用 `twlegalrag search`，以預設 `-n 5` 快速確認結果是否聚焦在正確爭點。

> 🔴 **CHECKPOINT — 方向確認**：展示 search 結果摘要（裁判字號、法院層級、爭點命中情形），讓使用者確認檢索方向正確，再繼續執行 Step 4 以後的步驟。如果結果偏離爭點，先調整查詢詞重新檢索。

4. 依 §法院優先順序規則 檢視結果，優先審閱高審級考察重點。
5. 若需要後續引用、分析或交付，使用 `twlegalrag pack` 產生判決封包 JSON；原則上先用 `-n 6`，必要時再逐步增加。
6. 讀取封包內容，擷取每筆裁判的 `citation_id`、裁判字號、法院層級與關鍵片段。
7. 依 §法院優先順序規則 整理裁判，優先處理最高院級裁判。
8. 僅使用封包中的裁判資料，作為後續書狀或法律分析的論證依據。
9. 要求下游生成內容以 `[J1]`、`[J2]` 等格式標註引用。
10. 將生成的回答、分析或書狀寫入暫存文字檔。
11. 使用 `python scripts/check.py <bundle.json> <answer.txt>` 執行引用檢查。

> 🔴 **CHECKPOINT — 引用驗證確認**：展示引用檢查結果（哪些通過、哪些需修正）。只有在使用者確認引用無誤後，才交付最終結果。若有引用不一致，先修正並重新執行 Step 11，不得跳過。

12. 只有在引用檢查完成且使用者確認後，才交付最終結果。

## scripts

### `scripts/check.py`

此腳本用於包裝 `twlegalrag check`，統一執行引用檢查流程。

用途：

- 檢查 bundle 檔案與回答檔案是否存在。
- 呼叫 `twlegalrag check`。
- 直接回傳原始檢查結果與退出碼，供上游流程判斷是否通過。

建議用法：

```bash
python scripts/check.py <bundle.json> <answer.txt>
```

## 輸出要求

使用本技能輸出結果時，應盡量包含以下資訊：

- 本次檢索的法律爭點摘要。
- 實際使用的查詢詞。
- 重要裁判的 `citation_id`、裁判字號與法院名稱或層級。
- 每則重要裁判的簡短見解摘要或關鍵理由片段。
- 是否已執行引用檢查。
- 若引用檢查未通過，哪些標記需要修正，以及修正原因。

## 失敗處理

| 觸發條件 | 一線修復 | 仍失敗兜底 |
|----------|----------|----------|
| 檢索結果過少、過廣或偏離爭點 | 調整查詢詞，拆解或擴充關鍵字後重新 search | 拆分為多個較小子爭點，分批檢索後合併結果 |
| 最高法院裁判不足或無法支撐主要論證 | 補充臺灣高等法院及其分院裁判，不一開始就無差別擴大範圍 | 告知使用者目前資料庫可用裁判有限，討論是否降低審級要求或調整論證策略 |
| `scripts/check.py` 顯示引用不一致 | 依回報訊息修正對應的 `[Jn]` 標記，再重新執行 `scripts/check.py` | 若多次修正後仍不一致，重新執行 `pack` 更新 bundle，再從 Step 9 重新生成內容 |
| 使用者法律問題範圍過大 | 協助拆成較小、較可檢索與可驗證的子爭點，再逐步處理 | 若仍無法聚焦，請使用者提供具體案情或爭點後再繼續 |
| 環境無法執行 `twlegalrag` | 告知本技能依賴已安裝的 CLI，並提醒使用者先執行 `pip install twlegalrag` | 若安裝後仍失敗，確認 Python 環境與 PATH 設定；本技能不以 HTTP 請求或爬蟲繞過此依賴 |
| `twlegalrag search` 或 `pack` 逾時、回傳網路錯誤 | 等待 30 秒後重試一次 | 仍失敗則告知使用者檢索服務可能暫時不可用，建議稍後再試；不以舊資料或快取結果替代 |
| bundle JSON 損壞或格式不符 | 重新執行 `twlegalrag pack` 產生新的 bundle | 若仍異常，告知使用者並請其確認 CLI 版本（`twlegalrag --version`）是否相容 |

## 規則

**引用正確性（絕對禁止）：**

- 🔴 不得捏造裁判字號、裁判理由、裁判結論或引文內容。
- 🔴 不得引用未出現在 bundle 中的裁判作為已驗證依據。
- 🔴 不得把未經 `check` 驗證的引用宣稱為已驗證。
- 優先使用可追溯的封包片段，不要依賴模糊記憶進行法律摘要。
- 在多筆裁判可選時，優先引用法院層級較高的裁判。
- 保持引用可重現、可稽核、可回溯。

**操作流程反例（不要這樣做）：**

- 🔴 不要跳過 CHECKPOINT 直接 pack：search 結果未經使用者確認方向，就直接執行 `twlegalrag pack`。這會讓後續整理與引用建立在錯誤爭點上。
- 🔴 不要繞過依賴：在 `twlegalrag` 不可用時，改用 HTTP 請求、網頁爬取或其他替代方式檢索。本技能的可追溯性來自 CLI 的封包機制。
- ⚠️ 不要對同一爭點反覆 pack 超過 2 次：若兩次 pack 結果都不理想，先回到查詢詞設計，而不是繼續盲目增加 `-n`。
- ⚠️ 不要把 search 結果直接當引用依據：`search --read` 的輸出僅供確認方向，未納入 bundle 的裁判不能作為已驗證引用。
- ⚠️ 不要在引用檢查失敗後仍交付：`check` 回傳不一致時，不能以「大致正確」或「小問題」為由略過修正直接交付。

## 範例

### 範例 1

使用者需求：

找出台灣法院對於競業禁止違約金過高的實務見解。

預期行為：

- 建立與競業禁止、違約金過高相關的查詢詞。
- 先用 `twlegalrag health` 確認服務正常。
- 先用 `twlegalrag search "競業禁止 違約金 過高" -n 5 --read` 快速確認方向。
- 先檢視最高法院裁判，其次檢視臺灣高等法院及其分院裁判。
- 使用 `twlegalrag pack "競業禁止 違約金 過高" -o bundle.json -n 6` 產生判決封包。
- 擷取重要裁判及其 `citation_id`。
- 以可追溯的引用格式整理裁判見解。

### 範例 2

使用者需求：

幫我檢查這份答辯狀裡引用的判決有沒有對。

預期行為：

- 先確認 bundle 與答辯狀文字檔存在。
- 使用 `python scripts/check.py bundle.json answer.txt` 執行引用檢查。
- 回報哪些引用正確，哪些引用需要修正。
- 在引用未驗證完成前，不將內容視為最終定稿。
