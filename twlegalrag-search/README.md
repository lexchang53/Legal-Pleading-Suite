# twlegalrag-search

台灣法院裁判語義檢索與引用驗證技能。透過自然語言在 AI Agent 工具中提問，即可搜尋台灣法院裁判資料庫，取得可追溯、可驗證的判決引用依據，適用於訴訟書狀撰寫與法律意見書研究。

> 本技能建立於 [法律偵探 (@aa0101181514)](https://github.com/aa0101181514) 所開發的開源工具 **[tw-legal-rag](https://github.com/aa0101181514/tw-legal-rag)** 之上。tw-legal-rag 是一套以語義搜尋技術為核心的台灣法院裁判 CLI 工具，本技能扮演其 AI Agent 整合層，負責查詢詞設計、流程控管、審級排序與引用驗證等環節。特別感謝作者的開源貢獻。

## 適用情境

**民事與刑事案件皆可使用。**

主技能套件（Legal-Pleading-Suite）的書狀撰寫模組（`draft-pleading`）目前以民事事件為主要設計對象；但 `twlegalrag-search` 作為獨立的裁判檢索技能，**不限案件類型**。無論是民事訴訟（給付之訴、確認之訴、形成之訴）或刑事案件（無罪答辯、量刑主張、程序爭議），均可透過本技能在 AI Agent 中以自然語言提問，由 AI 自動建構查詢詞、檢索相關裁判、整理成可引用的封包，供後續書狀或法律意見使用。

典型使用方式，例如：

```text
幫我查詢台灣法院關於刑事詐欺罪既遂與未遂認定的實務見解。
```

```text
請找出勞資爭議中，法院認定雇主違法解僱的相關案例。
```

## 為什麼需要這個技能？

書狀或法律意見書中的裁判引用，一旦字號錯誤、理由錯誤或根本不存在，不但無法支撐論點，還可能削弱整份文件的可信度。現有的 AI 工具在這一點上有明確的侷限：它可以生成看起來合理的裁判字號，但不保證該裁判確實存在。

為了杜絕 AI 憑空捏造裁判，本技能的核心邏輯是：**透過「打包封包」把真實裁判下載並鎖定，作為 AI 寫作時引用的唯一依據**。
其流程為：**先用真實資料庫檢索 → 打包成固定的封包 → 讓 AI 對照寫作 → 驗證引用**，確保最終文件引用的每一字號皆真實可查。

## 前置需求

本技能依賴 `twlegalrag` CLI，使用前需先安裝：

```bash
pip install twlegalrag
```

若指令找不到或回傳服務異常，技能不會以其他方式替代檢索——保持可追溯性比便利性更重要。

## 運作模式概覽

這個技能採三段式流程：

```
查詢預覽（search）→ 打包封包（pack）→ 驗證引用（check）
```

**1. 查詢預覽**  
先用 `search` 確認檢索方向，而不是一開始就打包所有資料。目的是讓使用者（或 AI）先確認結果真的聚焦在正確爭點，避免後續整理了大量不相關裁判。

**2. 打包封包**  
將篩選後的真實裁判下載並固化為一個 `bundle.json` 檔案。**這是為 AI 框定的「寫作真實邊界」**——AI 後續只能使用封包內的判決（如 `[J1]`、`[J2]`），徹底防範 AI 憑記憶捏造字號。

**3. 驗證引用**  
書狀或分析完成後，透過 `scripts/check.py` 核對文中的 `[J1]`、`[J2]` 等引用標記是否與封包內容相符。這一步不是選配，是交付前的必要關卡。

## 關於查詢詞的一個重要觀念

臺灣民法中外觀相近但效果不同的概念（例如終止、解除、撤銷、無效），在實務上有截然不同的裁判走向。若將其混用作為查詢詞擴展，很可能檢索到完全不同方向的裁判，反而誤導書狀論述。

因此，本技能明確要求：
- 查詢詞只允許同一概念的書寫變體擴展（如「條款」↔「約款」）。
- 不同法律效果的概念（終止 ≠ 解除、解除 ≠ 撤銷、無效 ≠ 撤銷），即便在日常語境中看起來相近，也絕對不得互換使用於查詢詞中。

同時，技能強制要求正反雙向查證——不能只收集「支持己方」的裁判，必須同時主動檢索法院「駁回」或「不予准許」的反向案例，讓摘要具備完整的訴訟風險評估。

## 目錄結構

```
twlegalrag-search/
├── SKILL.md           ← 給 AI Agent 讀的執行規格，包含完整指令、流程與規則
├── README.md          ← 本說明文件（給人看）
├── scripts/
│   └── check.py      ← 引用驗證包裝腳本
└── test-prompts.json  ← 測試用查詢範例
```

### `scripts/check.py`

這個腳本是 `twlegalrag check` 的包裝器，加入了前置驗證（bundle 與回答檔案是否存在）後再呼叫底層指令。整個技能的引用檢查統一透過此腳本執行。

```bash
python scripts/check.py <bundle.json> <answer.txt>
```

## 與其他模組的關係

- **`draft-pleading`**：書狀撰寫模組。在進入草稿生成階段前，可先用 `twlegalrag-search` 取得可引用的封包，再由 `draft-pleading` 生成帶有 `[J1]`、`[J2]` 引用格式的書狀草稿。
- **`legal-opinion`**：法律意見書模組。同樣可搭配此技能，取代以往依賴網路搜尋或模糊記憶的裁判引用方式。
- **`notebooklm-skill`**：兩者可互補——NotebookLM 適合查詢使用者自行整理的案件知識庫；`twlegalrag-search` 適合從台灣法院裁判資料庫取得可驗證的實務見解。

## 常見問題

**Q：和直接讓 AI 引用記憶中的判決有什麼差別？**  
A：直接讓 AI 寫判決通常會產生「字號不存在」或「張冠李戴」的幻覺。而「打包封包」是強制把真實判決先下載保存起來，就像給 AI 一本只包含真實判決的「開卷參考書」，AI 只能從裡面抄，不能自己編。

**Q：刑事案件也可以用嗎？**  
A：可以。tw-legal-rag 的語義搜尋範圍涵蓋民事與刑事裁判，不限案件類型。

**Q：可以讓 AI 自動擴展查詢詞嗎？**  
A：可以，但本技能對擴展範圍有明確限制，AI 在臺灣法律語境下不得將終止、解除、撤銷、無效等不同效果的概念互換使用，以避免檢索出方向相反的裁判。
