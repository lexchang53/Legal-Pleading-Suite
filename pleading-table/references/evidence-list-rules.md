# 高院證據清單表 — 欄位規則與 AI 工作流程

本文件定義高院證據清單表的所有欄位規則，供 `build_issue_table.py` 與 AI 撰稿時參考。

> [!IMPORTANT]
> 本文件定義的「高院證據清單表」專屬於**高等法院案件**，與 `draft-pleading` 中由司法院規範的「證據或文件清單」**完全不同**，不得混用。

---

## 表格欄位定義

| 序號 | 欄頭（上列） | 欄頭（下列） | JSON 欄位名稱 | 說明 |
|------|------------|------------|-------------|------|
| TC0 | 序號 | （同上） | 由腳本自動生成，AI 不填 | 依時間排序後的流水號 |
| TC1 | 證據時間 | （同上） | `evidence_date` | 民國紀元壓縮格式 |
| TC2 | 證據名稱 | （同上） | `evidence_name` | 一望即知之簡稱 |
| TC3 | 證據簡要內容 | （同上） | `evidence_summary` | 不超過 20 字 |
| TC4 | 待證事實 | （同上） | `fact_to_prove` | 簡要待證事實 |
| TC5 | 證據附卷位置 | 證據編號 | `evidence_code` | 當事人原始編號 |
| TC6 | 證據附卷位置 | 法院卷宗頁碼 | `court_page` | 依代碼體系填寫 |
| TC7 | 備註意見 | （同上） | `remarks` | 選填，預設空白 |

---

## 欄位規則詳述

### 1. 序號（TC0）

- 由腳本依陣列索引自動生成（1, 2, 3…）。
- AI **不得**在 JSON payload 中填入。
- 此欄與「證據編號」（TC5）完全無關，請勿混淆。

### 2. 證據時間（TC1）

- **格式**：民國紀元壓縮格式，7 位數字，依序為**年份（3位）+ 月份（2位）+ 日期（2位）**。
  - 範例：民國 111 年 3 月 10 日 → `1110310`
  - 範例：民國 109 年 3 月 19 日 → `1090319`
- **無法確認時**：填 `0000000`（7 個零）。
- **排序作用**：整份清單**必須依證據時間由小到大排序**（最早發生在前）。填 `0000000` 的項目統一排在最後。

> [!CAUTION]
> **修改後強制重新排序**：當使用者修改任何項目的「證據時間」後，AI **必須**重新依 `evidence_date` 排序整份清單並重新生成草稿（含更新後的序號）。禁止僅局部修改而不重新排序。

## 樣式與排版規範

為確保與 `table-tmpl.docx` 完美契合，系統強制使用以下樣式：

- **文件標題**（例如「被上訴人證據清單表」）：套用「**書狀_標題**」（20pt 標楷體）。
- **表格標題列**（列 0、列 1）：套用「**爭點表_標題**」。
- **表格資料列**（列 2 以下）：套用「**爭點表_內容**」。
- **跨頁處理**：標題列（列 0、列 1）必須設定為 `tblHeader`，確保跨頁時重複標頭。

---

### 3. 證據名稱（TC2）

- 以一望即知之簡稱為佳（如「房屋租賃契約」「勞保投保資料」）。

### 4. 證據簡要內容（TC3）

- 先接續「證據名稱」的補充說明，再記載與本案原因事實相關的重要內容。
- **不超過 20 個字**為原則。

### 5. 待證事實（TC4）

- 簡要記載提出該項證據所欲證明之事實。

### 6. 證據編號（TC5）

> [!IMPORTANT]
> **不採用**高院規範中的壓縮代碼（甲、甲1、甲A1 等）。**直接沿用**當事人在書狀中使用的原始證據編號。

- 原告一審：`原證1`、`原證2`…
- 被告一審：`被證1`…
- 上訴人二審：`上證1`…（若一審為甲/乙證體系則續接）
- 被上訴人：`被上證1`…
- **法院調查**：無論依職權或依聲請，一律填「**法院**」（不附編號）。

### 7. 法院卷宗頁碼（TC6）

採統一壓縮代碼格式。

**僅發回一次案件（更一審）**：

| 代碼 | 指涉 | 範例 |
|------|------|------|
| 原 | 一審卷 | `原一8`（原審卷一第8頁） |
| 前 | 前審卷（重上字） | `前三208` |
| 本 | 本院本案卷 | `本一15` |

**多次發回案件（更二審以上）**：以原始案號之「字別」取代「更0」

| 代碼 | 指涉 | 範例 |
|------|------|------|
| 原 | 一審卷 | `原一8` |
| [字別] | 原始上訴審卷（依案號） | `重上四98`、`重勞上三56` |
| 更Ⅰ | 重上更一字卷 | `更Ⅰ二58` |
| 更Ⅱ | 重上更二字卷 | `更Ⅱ三56` |
| 本 | 本院本案卷 | `本一15` |

> [!NOTE]
> 僅發回一次時，「前」可繼續使用；律師若偏好精確對應，亦可直接以字別代碼（如「重上」）標示。

### 8. 備註意見（TC7）

- 模板保留欄位，維持高院格式完整性。
- **預設不填寫**，儲存格留白。
- 僅在使用者明確指示時，填入對該證據「形式上真正爭執與否」。

---

## 證據專屬過濾規則

> [!CAUTION]
> 列入高院證據清單表的項目，**僅限**證據編號中含有「**證**」字的項目（如甲證、乙證、原證、被證、上證、被上證等）。
>
> 凡是在書狀狀尾以「**附件**」標示的項目（如法院裁判書、學術文獻等），**絕對不得列入**高院證據清單表。

---

## AI 自動掃描與草稿確認工作流程

當使用者要求製作「高院證據清單」時，AI **必須**依以下步驟執行：

1. **掃描工作目錄**：搜尋同資料夾內的所有既有書狀（優先 `.md`，次為 `.docx`、`.odt`），提取每份書狀中的所有證據引用與證據清單。
2. **過濾與彙整**：依「證據專屬過濾規則」，僅保留含「證」字的編號，排除所有「附件」項目。
3. **依時間排序**：將所有証據依 `evidence_date` 由小到大排列，`0000000` 排在最後。
4. **形成草稿清單**：以有序列表格式呈現給使用者，每項必須包含：
   - 證據時間（壓縮格式）
   - 證據名稱
   - 證據簡要內容
   - 待證事實
   - 證據編號
   - 法院卷宗頁碼
5. **等待使用者確認**：使用者審閱草稿並告知哪些列入、哪些刪除或修改後，AI 才將確認後的清單轉為 JSON payload 的 `evidence_list`。

> [!IMPORTANT]
> AI **絕對禁止**在未經使用者確認的情況下，直接將掃描結果寫入 JSON payload 或產出 DOCX。

## JSON Payload 欄位格式

`evidence_list` 為頂層選填欄位：

```json
{
  "evidence_list": {
    "title": "被上訴人證據清單表",
    "items": [
      {
        "evidence_date": "1061102",
        "evidence_name": "房屋租賃契約",
        "evidence_summary": "16號2樓前次租約，租期106/11/5至108/11/4",
        "fact_to_prove": "房屋實由詹吉雄親自出租管理",
        "evidence_code": "原證7",
        "court_page": "原一85-87",
        "remarks": ""
      }
    ]
  }
}
```

### 欄位驗證規則

| 欄位 | 必填 | 驗證規則 |
|------|------|---------|
| `title` | ✅ | 格式為「[稱謂]證據清單表」 |
| `items` | ✅ | 陣列，非空；省略或空陣列時不產出表格 |
| `evidence_date` | ✅ | 7 位數字字串（`0000000`~`9999999`） |
| `evidence_name` | ✅ | 非空字串 |
| `evidence_summary` | ✅ | 非空字串，建議不超過 20 字 |
| `fact_to_prove` | ✅ | 非空字串 |
| `evidence_code` | ✅ | 必須含「證」字，或為「法院」；不得為「附件」 |
| `court_page` | ✅ | 依代碼體系填寫 |
| `remarks` | ❌ 選填 | 省略或空字串時儲存格留白 |

---

## 主書狀狀尾連動規則

當使用者要求製作高院證據清單表時，主書狀（答辯狀、上訴理由狀、陳報狀、爭點整理狀等）的狀尾「證據或文件清單：」區塊末尾，**必須加上**：

```
附件：證據清單
```

此行**不佔用**證據編號（不編為甲證N、乙證N 等），以「附件」前綴處理。
