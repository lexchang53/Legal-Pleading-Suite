# ⚖️ AI 法律訴訟書狀自動化套件  
[![License: CC BY-NC-SA 4.0](https://img.shields.io/badge/license-CC--BY--NC--SA--4.0-lightgrey.svg)](LICENSE.md)
![Monorepo](https://img.shields.io/badge/structure-monorepo-blue)
![Taiwan Legal Workflow](https://img.shields.io/badge/focus-Taiwan%20Legal%20Workflow-2f855a)

本專案面向台灣法律實務工作者（律師、法務、研究者），整合 AI 書狀撰寫、NotebookLM 知識檢索、表格排版、ODT 轉檔與法律結構可視化，形成一套可重複使用的訴訟文件工作流。

它的重點不是單一 prompt，而是把「案件資料整理 → 撰寫計畫 → 知識檢索 → 草稿確認 → 最終排版」串成一條可審閱、可校正、可延續的工作管線。

> [!WARNING]
> 本專案是法律工作輔助工具，不替代律師或使用者的專業判斷。任何草稿、書狀、附表、圖表或轉檔結果，在對外使用或提交法院前，仍應由使用者自行核對案件事實、法律依據、裁判引用、證據標號、頁碼與格式。

## 專案特色

- **書狀工作流導向**：不是只產生文字，而是依序處理案件資料、撰寫計畫、草稿確認與最終排版。
- **NotebookLM 整合**：可查詢使用者上傳至 NotebookLM 的筆記本，補強法律見解與案件資料檢索。
- **表格型法律文件支援**：可輸出爭點整理表、聲請調查證據表與高院證據清單表。
- **DOCX / ODT 雙路徑**：以 DOCX 為主輸出，必要時可轉為 ODT 供 LibreOffice 工作流使用。
- **法律結構可視化**：可生成當事人關係圖、證據鏈分析圖、請求權基礎圖與程序流程圖。
- **Monorepo 整合**：將撰狀、表格、轉檔、知識檢索與視覺化模組集中管理，便於維護與擴充。

## 適用情境

本專案特別適合以下工作場景：

- 撰寫民事起訴狀、答辯狀、準備書狀、上訴理由狀。
- 整理爭點、證據鏈與既有卷證，並延續同案既有書狀脈絡。
- 需要把 NotebookLM 作為案件資料庫、裁判檢索來源或知識庫使用。
- 需要輸出高院格式證據清單、爭點整理表或調查證據表。
- 需要把複雜法律關係轉為圖解，作為開庭、簡報或內部分析材料。

## 核心模組

### 1. `draft-pleading/`
核心撰狀引擎。負責讀取案件資料、分析爭點、形成撰寫計畫、視需要查詢知識庫、生成 Markdown 草稿，並在草稿確認後交由內部排版流程輸出最終 DOCX。

支援的典型書狀包括：

- 民事起訴狀
- 答辯狀
- 準備書狀
- 上訴理由狀
- 爭點整理狀
- 聲請調查證據狀

### 2. `pleading-table/`
處理需要精確 Word 表格排版的法律文件，包含：

- 爭點整理表
- 聲請調查證據表
- 高院證據清單表

### 3. `docx-to-odt/`
在有明確需求時，將 DOCX 轉換為 ODT，供 LibreOffice 工作流繼續編修。

### 4. `notebooklm-skill/`
整合 Google NotebookLM 查詢能力。此目錄名稱為 `notebooklm-skill`，其內部技能名稱為 `notebooklm`。可用於管理 notebook library、驗證登入狀態、指定 notebook 或 URL 查詢，以及延伸追問補查。

### 5. `legal-architecture/`
法律結構可視化模組。輸出為自包含 HTML 檔案（內嵌 SVG），適合在瀏覽器開啟、列印，或截圖嵌入 Word / PPT。

## 快速開始

### 先備條件

建議您至少具備以下環境：

- Python 3.10+（或依各模組實際需求版本）
- Git（選配，若不熟悉指令可直接下載 ZIP 檔）
- LibreOffice（若您需要 ODT 轉檔與後續編修）
- 可登入 Google 帳號的瀏覽器環境（若您要使用 NotebookLM）

### 1. 下載專案

對於熟悉程式開發的使用者，建議使用 `git` 以便日後同步更新：

```bash
git clone https://github.com/lexchang53/Legal-Pleading-Suite.git
cd Legal-Pleading-Suite
```

**【法律專業人士建議方式】**：
若您不熟悉 `git` 指令，可以直接在 GitHub 頁面點擊綠色的 **「Code」** 按鈕，選擇 **「Download ZIP」**。下載後解壓縮，即可直接使用其中的檔案，完全不需要安裝 Git。

### 2. 檢查授權與模組

- 閱讀根目錄 `LICENSE.md`
- 閱讀第三方模組各自的授權與原始專案說明
- 確認 `draft-pleading/`、`notebooklm-skill/`、`legal-architecture/` 等子模組已完整下載

### 3. 準備案件工作資料夾

在您的實際工作區中，建議採用「一案一資料夾」方式，並放入：

- `案件基本資料.txt`
- 會談逐字稿或案情摘要
- 既有書狀
- 證據資料
- 筆錄
- 裁判
- 其他案件相關文件

### 4. NotebookLM 初次使用（如需）

`notebooklm-skill` 建議先完成登入驗證與 notebook 檢查。依其設計，應透過 `run.py` 包裝器執行，而不是直接呼叫子腳本。

```bash
cd notebooklm-skill
python scripts/run.py auth_manager.py status
python scripts/run.py auth_manager.py setup
python scripts/run.py notebook_manager.py list
```

若您尚未把 notebook 加入 library，可再依需要新增：

```bash
python scripts/run.py notebook_manager.py add \
  --url "https://notebooklm.google.com/notebook/..." \
  --name "我的案件筆記本" \
  --description "本筆記本包含勞資爭議案件之雙方書狀、證據、筆錄與裁判" \
  --topics "勞動法,資遣費,確認僱傭關係"
```

### 5. 進一步深入瞭解

本專案將複雜的排版邏輯與法律撰稿規則抽離至獨立文件。若您想更深入瞭解內部的運作準則，請參考以下目錄內的 `.md` 檔案：

- **`draft-pleading/references/`**：包含草稿撰寫規則 (`phase5-draft-rules.md`)、知識庫查詢流程、撰寫計畫要求等。
- **`pleading-table/references/`**：包含爭點整理表的詳細排版與自動編號細則。

## 典型工作流程

本套件採取偏向「人機協作式自動化」的設計，而不是直接一鍵吐出定稿。

1. **讀取案件資料**  
   讀取工作目錄中的卷證、既有書狀、訪談紀錄、筆錄、裁判或其他資料。

2. **提出撰寫計畫**  
   先整理主張架構、爭點與證據鏈，供使用者確認。

3. **知識檢索與補強**  
   視案件需要查詢 NotebookLM 與其他來源，補強法律見解、案件資料或爭點分析。

4. **生成 Markdown 草稿**  
   先輸出可審閱、可修改的書狀草稿，而不是直接跳到最終檔。

5. **確認後排版輸出**  
   草稿確認後，再生成 DOCX；若使用者另有要求，則接續轉為 ODT，或交由其他模組處理附表。

6. **視需要生成法律結構圖**  
   在多人關係、複雜證據鏈或請求權基礎分析案件中，可再搭配 `legal-architecture` 產出圖解。

## 常見使用方式

一般情況下，使用者不需要把所有中間步驟都寫進 prompt，因為技能本身就會依工作流先提計畫、再出草稿、最後排版。

### 一般撰狀

```text
請先讀取目前工作目錄中的案件資料，幫我撰寫一份民事起訴狀。
如有需要，也請參考我指定的 NotebookLM 筆記本。
```

### 爭點整理狀（含表格）

```text
請幫我撰寫一份附表格的爭點整理狀。
```

### 另附高院格式證據清單

```text
請另外附上高院格式證據清單。
```

### 最後轉成 ODT

```text
最後請幫我轉成 ODT 檔。
```

### 生成法律可視化圖

```text
請畫出本案的當事人關係圖與證據鏈分析圖。
```

## 書狀排版規格與範本

本套件的最終排版輸出高度依賴預設的 Word 範本檔。

### 1. 範本檔案位置
- **`draft-pleading/assets/pleading-tmpl.docx`**

### 2. 核心排版規格
- **字型與樣式**：全文預設使用 **標楷體 (BiauKai)**，標題與內文均預設符合台灣法院實務規格。
- **行距與段落**：採用標準行距（約 1.5 倍）與懸掛縮排，確保自動編號後的法條與論點對齊美觀。
- **行編號 (Line Numbering)**：
  - **DOCX 輸出**：行編號固定顯示於頁面 **左側**。
  - **ODT 轉換後**：若經由 `docx-to-odt` 處理，行編號會自動調整為頁面 **外側 (Outside)**，符合雙面列印或特定格式需求。
- **表格規格 (高院格式)**：
  - 爭點整理表與證據清單表參考 **臺灣高等法院** 的標準格式（並根據 AI 解析需求稍作優化）。
  - 支援 **跨頁重複標題列**，確保長表格在翻頁後仍能清楚辨識欄位。
  - 表格內文字字型大小與框線粗細均已預先調校。

> [!TIP]
> 使用者可以依照事務所的偏好，直接開啟 `pleading-tmpl.docx` 調整字型、行高或邊界設定。只要不刪除內部的樣式名稱，排版引擎就能自動套用您的自訂格式。

## 案件資料準備

### 一案一資料夾
建議每個案件使用一個獨立工作資料夾，將相關卷證、既有書狀、筆錄、裁判與訪談資料集中管理，以降低資料混淆風險，並利於延續既有證據編號與卷宗頁碼。

### 優先使用 Markdown
雖然系統可透過安裝對應技能（Skills）支援讀取 **`.pdf`**（需 `pdf` 技能）、**`.docx`**（需 `docx` 技能）、**`.odt`**（需 `writer` 技能）等格式，但若條件允許，仍強烈建議將案件資料盡量轉為 **Markdown (.md)** 或 **純文字 (.txt)**。這類純文字格式可被 AI 直接、高速且精準地讀取，完全不需依賴額外解析工具。

> [!TIP]
> 若您的系統環境已安裝全域工具 **Pandoc**，將能大幅提升 AI 對於各類文件格式的轉換成功率與讀取效能。

### `案件基本資料.txt` 範例

建議在工作資料夾中建立一份 `案件基本資料.txt`，至少整理如下資訊：

```text
原告：王小明
地址：新北市板橋區○○路100號
電話：02-1234-5678
電子郵件：plaintiff@example.com

被告：大安股份有限公司
地址：臺北市大安區○○路200號
電話：02-2345-6789

法院：臺灣新北地方法院
案由：給付資遣費
案號：可留白；如已有案號可填入
```

案件基本資料越完整，越有助於整理狀首欄位、法院資訊、當事人稱謂與案由脈絡；但最終內容仍應由使用者自行核對。

### 會談逐字稿與案件材料
若您平常會先與當事人討論案情，再將錄音轉成逐字稿，也可以直接把逐字稿放入工作區，讓系統據以整理事實經過與初步主張。

## NotebookLM 整合

本專案中的 NotebookLM 並不只用來查詢裁判見解。實務上，使用者也可以把整個案件相關材料整理到自訂的 NotebookLM 筆記本中，再由系統輔助檢索與吸收。

可放入 NotebookLM 的材料包括：

- 雙方書狀
- 證據資料
- 會談紀錄或逐字稿
- 開庭筆錄
- 法院裁判
- 法規、函釋、研究資料或自行整理的知識筆記

換句話說，NotebookLM 在本專案中不只是「裁判檢索工具」，也可以扮演案件資料庫與法律知識庫的雙重角色。

## 法律可視化

`legal-architecture` 為專門的法律圖解技能，輸出為單一 HTML 檔案，內嵌 SVG，不需額外腳本即可在現代瀏覽器中開啟。

### 適用情境

- 當事人關係圖
- 證據鏈分析圖
- 請求權基礎分析圖
- 訴訟程序流程圖
- 契約關係圖
- 法律體系層級圖

### 特點

- 自包含 HTML，方便保存、傳閱與列印
- 針對中文長字串做斷行處理
- 採法律語意配色，區分當事人、證據、法律規範、程序節點、爭議焦點與裁判結果
