# ⚖️ AI 法律訴訟書狀自動化套件  
[![License: CC BY-NC-SA 4.0](https://img.shields.io/badge/license-CC--BY--NC--SA--4.0-lightgrey.svg)](LICENSE.md)
![Monorepo](https://img.shields.io/badge/structure-monorepo-blue)
![Taiwan Legal Workflow](https://img.shields.io/badge/focus-Taiwan%20Legal%20Workflow-2f855a)

本專案面向台灣法律實務工作者（律師、法務、研究者），整合 AI 書狀撰寫、NotebookLM 知識檢索、表格排版、ODT 轉檔與法律結構可視化，形成一套可重複使用的訴訟文件工作流。

它的重點不是單一 prompt，而是把「案件資料整理 → 撰寫計畫 → 知識檢索 → 草稿確認 → 最終排版」串成一條可審閱、可校正、可延續的工作管線。

> [!WARNING]
> 本專案是法律工作輔助工具，不替代律師或使用者的專業判斷。任何草稿、書狀、附表、圖表或轉檔結果，在對外使用或提交法院前，仍應由使用者自行核對案件事實、法律依據、裁判引用、證據標號、頁碼與格式。

## 專案特色

- **書狀工作流導向**：不是只產生文字，而是依序處理案件資料、撰寫計畫、草稿確認與最終排版。
- **NotebookLM 整合**：可查詢使用者上傳至 NotebookLM 的筆記本，補強法律見解與案件資料檢索。
- **表格型法律文件支援**：可輸出爭點整理表、聲請調查證據表與高院證據清單表。
- **DOCX / ODT 雙路徑**：以 DOCX 為主輸出，必要時可轉為 ODT 供 LibreOffice 工作流使用。
- **法律結構可視化**：可生成當事人關係圖、證據鏈分析圖、請求權基礎圖與程序流程圖。
- **Monorepo 整合**：將撰狀、表格、轉檔、知識檢索與視覺化模組集中管理，便於維護與擴充。

## 適用情境

本專案特別適合以下工作場景：

- 撰寫民事起訴狀、答辯狀、準備書狀、上訴理由狀。
- 整理爭點、證據鏈與既有卷證，並延續同案既有書狀脈絡。
- 需要把 NotebookLM 作為案件資料庫、裁判檢索來源或知識庫使用。
- 需要輸出高院格式證據清單、爭點整理表或調查證據表。
- 需要把複雜法律關係轉為圖解，作為開庭、簡報或內部分析材料。

## 核心模組

### 1. `draft-pleading/`
核心撰狀引擎。負責讀取案件資料、分析爭點、形成撰寫計畫、視需要查詢知識庫、生成 Markdown 草稿，並在草稿確認後交由內部排版流程輸出最終 DOCX。

在處理台灣勞動法相關案件時，`draft-pleading` 預設搭配下列 NotebookLM 筆記本作為優先查詢來源之一：

- [台灣勞動法知識庫｜張清浩律師編選](https://notebooklm.google.com/notebook/cba7ccc1-4bbd-4329-8bde-6a6e7a71a857)

支援的典型書狀包括：

- 民事起訴狀
- 答辯狀
- 準備書狀
- 上訴理由狀
- 爭點整理狀
- 聲請調查證據狀

### 2. `pleading-table/`
處理需要精確 Word 表格排版的法律文件，包含：

- 爭點整理表
- 聲請調查證據表
- 高院證據清單表

### 3. `docx-to-odt/`
在有明確需求時，將 DOCX 轉換為 ODT，供 LibreOffice 工作流繼續編修。

### 4. `notebooklm-skill/`
整合 Google NotebookLM 查詢能力。此目錄名稱為 `notebooklm-skill`，其內部技能名稱為 `notebooklm`。可用於管理 notebook library、驗證登入狀態、指定 notebook 或 URL 查詢，以及延伸追問補查。

### 5. `legal-architecture/`
法律結構可視化模組。輸出為自包含 HTML 檔案（內嵌 SVG），適合在瀏覽器開啟、列印，或截圖嵌入 Word / PPT。

## 環境安裝

### 先備條件

建議您至少具備以下環境：

- 具 **Agentic 能力** 的 AI Agent 或 AI Coding Assistant（推薦如 Google Antigravity、Codex app、Claude Code），且以具有 IDE 或類 IDE 工作介面者為佳
- Python 3.10+（或依各模組實際需求版本）
- Git（選配，若不熟悉指令可直接下載 ZIP 檔）
- Microsoft Word（或相容之文書處理軟體，如 LibreOffice Writer）
- LibreOffice（若您需要轉檔為 ODT）
- 可登入 Google 帳號的瀏覽器環境（若您要使用 NotebookLM）

> [!IMPORTANT]
> 本專案不是單純提供一段 prompt 使用，而是由多個技能模組共同構成的工作流。
> 因此，您所使用的 AI 工具最好能夠：
> - 載入技能（Skills）
> - 偵測各技能目錄中的 `SKILL.md`
> - 讀取與操作工作目錄中的案件資料
> - 依工作流分階段執行任務（如先提出撰寫計畫，再生成草稿與排版輸出）

### 1. 下載專案

對於熟悉程式開發的使用者，建議使用 `git` 以便日後同步更新：

```bash
git clone https://github.com/lexchang53/Legal-Pleading-Suite.git
cd Legal-Pleading-Suite
```

若您不熟悉 `git` 指令，可以直接在 GitHub 頁面點擊綠色的 **「Code」** 按鈕，選擇 **「Download ZIP」**。下載後解壓縮，即可直接使用其中的檔案，完全不需要安裝 Git。

### 2. 技能安裝關鍵說明

> [!WARNING]
> **不要**將整個 `Legal-Pleading-Suite/` 母目錄直接放入 AI 工具的技能區（Skills folder）。
> 多數 AI 工具只會偵測技能區中「直接可見」的技能目錄與其中的 `SKILL.md`，若您只是把整個解壓後的母目錄原封不動放進去，AI 工具通常**無法正確辨識**位於深層子目錄內的技能。

本專案採 Monorepo 結構管理，但真正應加入 AI 工具技能區的，是以下 **5 個核心技能子目錄**：

- `draft-pleading/`
- `pleading-table/`
- `docx-to-odt/`
- `notebooklm-skill/`
- `legal-architecture/`

也就是說，正確做法通常是：

1. 先下載或解壓縮整個 `Legal-Pleading-Suite/`
2. 打開該目錄
3. 找出上述 5 個子目錄
4. 將這 5 個子目錄個別移入您的 AI 工具技能區，或依工具支援方式逐一加入
5. 確認每個技能目錄底下都可直接看到 `SKILL.md`

錯誤示意（不要這樣做）：

- 直接把 `Legal-Pleading-Suite/` 整個資料夾丟進技能區

正確示意（建議這樣做）：

- 技能區內直接放入：
  - `draft-pleading/`
  - `pleading-table/`
  - `docx-to-odt/`
  - `notebooklm-skill/`
  - `legal-architecture/`

### 3. 非技能檔案應放哪裡？

下列內容**不是技能**，不需要放進 AI 工具的技能區：

- `README.md`
- `LICENSE.md`
- `examples/`

建議做法如下：

- 將整個 `Legal-Pleading-Suite/` 保留在您自己的專案資料夾、下載資料夾或工作區中，作為說明文件與參考資源
- `README.md` 供您閱讀安裝方式、工作流與使用說明
- `examples/` 供您查看範例輸出與預期成果格式
- 真正要安裝到 AI 工具技能區的，只有前述 5 個技能子目錄

### 4. NotebookLM 初次使用（如需）

`notebooklm-skill` 建議先完成登入驗證與 notebook 檢查。依其設計，應透過 `run.py` 包裝器執行，而不是直接呼叫子腳本。

```bash
cd notebooklm-skill
python scripts/run.py auth_manager.py status
python scripts/run.py auth_manager.py setup
python scripts/run.py notebook_manager.py list
```

若您希望直接搭配本專案預設的勞動法知識庫使用，可優先加入下列 NotebookLM 筆記本：

- [台灣勞動法知識庫｜張清浩律師編選](https://notebooklm.google.com/notebook/cba7ccc1-4bbd-4329-8bde-6a6e7a71a857)

此筆記本可作為處理台灣勞動法相關案件時的預設知識來源之一，特別適合用於補強勞資爭議案件中的法條、實務見解與爭點整理。

若您尚未把 notebook 加入 library，可再依需要新增：

```bash
python scripts/run.py notebook_manager.py add \
  --url "https://notebooklm.google.com/notebook/..." \
  --name "我的案件筆記本" \
  --description "王〇明訴壞〇公司案件之兩造書狀、證據、法院筆錄、文書與裁判" \
  --topics "王〇明, 壞〇公司, 給付資遣費"
```

### 5. 進一步深入瞭解

本專案將複雜的排版邏輯與法律撰稿規則抽離至獨立文件。若您想更深入瞭解內部的運作準則，請參考以下目錄內的 `.md` 檔案：

- **`draft-pleading/references/`**：包含草稿撰寫規則 (`phase5-draft-rules.md`)、知識庫查詢流程、撰寫計畫要求等。
- **`pleading-table/references/`**：包含爭點整理表的詳細排版與自動編號細則。

## 實戰工作流

本套件採取偏向「人機協作式自動化」的設計，不是直接一鍵吐出定稿，而是依照書狀工作的實際順序，逐步完成資料整理、論點形成、見解驗證、草稿撰寫與最終排版。

### 步驟 1：案件資料準備

建議每個案件使用一個獨立工作資料夾，將相關卷證、既有書狀、筆錄、裁判、訪談資料與其他案件文件集中管理，以降低資料混淆風險，並利於延續既有證據編號與卷宗頁碼。

雖然系統可透過安裝對應技能（Skills）支援讀取 **`.pdf`**、**`.docx`**、**`.odt`** 等格式，但仍強烈建議將案件資料優先準備為 **Markdown (`.md`)** 或 **純文字 (`.txt`)**。這類純文字格式可被 AI 直接、高速且精準地讀取，通常也更利於後續整理與追問。

若案件資料原始格式為 PDF，建議先要求 AI 將 PDF 內容辨識並轉成 **Markdown (`.md`)** 或 **純文字 (`.txt`)** 後，再作為後續撰狀輸入資料。這項建議同樣適用於 **掃描圖像型 PDF** 與 **文字型 PDF**，因為先轉成可直接讀取的純文字格式，通常更有助於後續的事實整理、爭點分析、證據比對與書狀撰寫。

例如，您可以直接將 PDF 拖入對話框後輸入：

```text
請 OCR 並轉成 markdown
```

完成轉寫後，再進一步要求 AI 依案件資料生成撰寫計畫、法律見解驗證報告與書狀草稿，通常會比直接以 PDF 作為主要輸入更穩定。

> [!TIP]
> 若您的系統環境已安裝全域工具 **Pandoc**，通常能提升 AI 對於各類文件格式的轉換成功率與讀取效能。

#### `案件基本資料.txt`（選填）

如果您不希望 AI 直接讀取真實的當事人姓名、地址、電話或其他敏感資訊，可以**不建立** `案件基本資料.txt`。在這種情況下，AI 仍可先依書狀格式需求，自動生成帶有佔位符的當事人欄位內容，例如以「原告 [姓名] 住[地址]」或「被告 [法人名稱] 設[地址]、法定代理人 [姓名] 地址同上」這類形式先建立書狀骨架，待草稿完成後再由使用者自行補入真實資料。

這種做法適合先確認書狀架構、請求內容、事實經過與論證方向，而把真實個資留到最後階段再手動補齊，以降低敏感資料在前期處理流程中的暴露範圍。

若您願意提供較完整的案件基本資料，AI 通常能更順利地建立當事人欄位、法院名稱、案由、狀尾資訊與其他書狀基礎欄位；但是否提供，仍可依您的隱私考量與工作習慣自行決定。

例如，您可以建立如下內容：

```text
原告：王〇明
地址：新北市板橋區○○路100號
電話：02-1234-5678
電子郵件：plaintiff@example.com

被告：壞〇股份有限公司
地址：臺北市大安區○○路200號

法院：臺灣新北地方法院
案由：給付資遣費
案號：可留白；如已有案號可填入
```

案件基本資料越完整，越有助於 AI 讀取後建立案號、股別、當事人資訊、案由、狀尾法院資訊等事項；但最終內容仍應由使用者自行核對。

若您平常會先與當事人討論案情，再將錄音轉成逐字稿，也可以直接把逐字稿放入工作區，讓系統據以整理事實經過與初步主張。

### 步驟 2：啟動撰狀流程

一般情況下，使用者不需要把所有中間步驟都寫進 prompt，因為技能本身就會依工作流先提出計畫、再進行法律見解補強、最後輸出草稿與正式排版。

#### 常用起始指令

```text
請先讀取目前工作目錄中的案件資料，幫我撰寫一份民事起訴狀。
```

或：

```text
請撰寫民事起訴狀：原告在113年1月1日借給被告50萬元，約定113年12月31日清償，利息年息10%，原告並於同日匯款至被告指定之銀行帳戶，有借據為憑。被告未曾給付利息，也不返還本金。原告於114年11月1日寄發存證信函催告後，被告仍未清償。請據此起草民事起訴狀。
```

#### 階段性產出 1：撰寫計畫

系統會先整理主張架構、爭點、請求權基礎與證據配置，供使用者確認後再往下走。

您可以先參考 [`examples/撰寫計畫.md`](examples/撰寫計畫.md) ，瞭解這個階段通常會如何整理書狀名稱、訴之聲明、事實經過、論點大綱、請求權基礎與證據清單。

#### 階段性產出 2：法律見解驗證

在正式生成草稿前，系統可就核心爭點補強法條依據、實務見解與攻防方向，避免在尚未確認法律基礎前就直接進入定稿式撰寫。

您可以先參考 [`examples/法律見解驗證報告.md`](examples/法律見解驗證報告.md) ，瞭解驗證報告通常如何整理法律爭點、判決字號、見解摘要與攻防策略。

若 NotebookLM 筆記本中找不到可直接使用的材料，而外部網路搜尋僅找到判決大意、無法確認官方裁判原文時，建議 AI 在結果中明確標示，例如：

- 「（來源係外部網路搜尋大意，內容或不可靠，仍應以官方裁判書或法律資料庫查詢之見解並引用之）」
- 「（來源僅載大意，原文待核）」

#### 草稿確認與最終排版

完成計畫與見解驗證後，系統會先輸出可審閱、可修改的 Markdown 草稿；待內容確認後，再交由內部排版流程輸出正式 DOCX。

特別是涉及表格之文件，系統常會先以**清單／條列式**呈現內容供核對，確認後再轉化為對應的 Word 表格。若使用者另有需求，則可再接續轉為 ODT，或交由其他模組處理附表。

### 步驟 3：進階功能整合

#### NotebookLM 深度檢索

本專案中的 NotebookLM 並不只用來查詢裁判見解。實務上，使用者也可以把整個案件相關材料整理到自訂的 NotebookLM 筆記本中，再由系統輔助檢索與吸收。

可放入 NotebookLM 的材料包括：

- 雙方書狀
- 證據資料
- 會談紀錄或逐字稿
- 開庭筆錄
- 法院裁判
- 法規、函釋、研究資料或自行整理的知識筆記

換句話說，NotebookLM 在本專案中不只是「裁判檢索工具」，也可以扮演案件資料庫與法律知識庫的雙重角色。

#### 法律結構可視化

`legal-architecture` 為專門的法律圖解技能，輸出為單一 HTML 檔案，內嵌 SVG，不需額外腳本即可在現代瀏覽器中開啟。

適用情境包括：

- 當事人關係圖
- 證據鏈分析圖
- 請求權基礎分析圖
- 訴訟程序流程圖
- 契約關係圖
- 法律體系層級圖

其特點包括：

## 書狀排版規格與範本

本套件的最終排版輸出高度依賴預設的 Word 範本檔。使用者可直接開啟下列範本檔查看或調整：

- [`draft-pleading/assets/pleading-tmpl.docx`](draft-pleading/assets/pleading-tmpl.docx)

### 核心排版規格

- **字型與樣式**：全文預設使用 **標楷體**。
- **字型大小**：
  - 狀首書狀名稱：**20pt**
  - 正文內容：**14pt**
  - 頁碼、行編號、表格內文字：**12pt**
- **行距與段落**：內文行距固定為 **26pt**。
- **多層次結構**：系統支援 **四層級自動編號** 結構（`一、` → `(一)` → `1.` → `(1)`），並配合懸掛縮排，確保法條引用與論點層次分明。
- **標點符號控制**：嚴格遵守 **標點符號不懸尾**（不超出右側邊界）之規範，確保排版整齊專業。
- **行編號（Line Numbering）**：
  - **DOCX 輸出**：行編號固定顯示於頁面 **左側**
  - **ODT 轉換後**：若經由 `docx-to-odt` 處理，行編號會自動調整為頁面 **外側（Outside）**
- **表格規格（高院格式）**：
  - 爭點整理表與證據清單表參考 **臺灣高等法院** 的標準格式（並根據 AI 解析需求稍作優化）
  - 支援 **跨頁重複標題列**
  - 表格內文字字型大小與框線粗細均已預先調校

> [!TIP]
> 使用者可以依照事務所或個人偏好，直接開啟 `pleading-tmpl.docx` 調整字型、行高、邊界、段落樣式與多層次清單設定。只要不要刪除內部既有樣式名稱，排版引擎通常仍可套用您的自訂格式。
