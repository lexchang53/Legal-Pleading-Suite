# 階段 4：查詢知識庫（完整操作指引）

本文件包含知識庫查詢的完整步驟、外部搜尋決策邏輯、查詢優化策略與結構化輸出格式。AI 進入階段 4 時讀取此文件。

## 知識庫選擇

- 筆記本選擇順序：
  1. 若使用者在對話中有明確指定筆記本名稱或 ID，優先使用該筆記本。
  2. 若無指定，AI 必須讀取設定檔 `draft-pleading/.pleading_preferences.json`。
     - 若檔案存在，請讀取其中的 `default_notebooks` 陣列，這將是使用者習慣使用的**私有知識庫**（例如「私人勞動法知識庫」或「民事證據法及第三審上訴」等）。AI 應依據本案之「案由」與「爭點性質」，自主選定一個最適合的私有筆記本進行查詢。若該案牽涉多個領域，亦可分別對不同筆記本產生 `queries.json`。
  3. 若設定檔不存在（代表此為第一次使用或 GitHub 公開版），AI **必須主動詢問使用者**：
     「即將開始查詢 NotebookLM，請問您要使用預設的【台灣勞動法知識庫｜張清浩律師編選】，還是要提供您專屬的 NotebookLM 筆記本網址？（若不需要查詢，亦可要求跳過。**系統將記錄您的選擇，您日後隨時可以用文字指令直接要求我增加、刪除或更換筆記本設定。**）」
     - 若使用者選擇**預設筆記本**，使用網址：`https://notebooklm.google.com/notebook/cba7ccc1-4bbd-4329-8bde-6a6e7a71a857`（AI 應一併詢問是否寫入 `.pleading_preferences.json` 以便日後自動使用）。
     - 若使用者提供**專屬網址**，使用該網址（AI 應一併詢問是否將此網址寫入 `.pleading_preferences.json`，以便日後自動使用）。
     - 若使用者要求修改、增加或刪除知識庫，AI 必須直接讀寫 `.pleading_preferences.json` 內的 `default_notebooks` 陣列來滿其需求。
     - 若使用者選擇**跳過**，則跳過查詢，直接進入階段 5。
- 查詢前先執行 `python scripts/run.py notebook_manager.py list` 確認指定或預設的筆記本可用（`scripts/run.py` 指 notebooklm skill 目錄下的）

> [!IMPORTANT]
> **例外跳過查詢**：
> 1. 撰寫「爭點整理狀」→ **絕對跳過**
> 2. 單純「聲請調查證據狀」且使用者**未**特別要求查詢 → **預設跳過**
> 若跳過查詢，直接進入階段 5。

## 4.1 第一層：內部知識庫主查詢（使用 batch_query.py）

> [!TIP]
> 使用 `scripts/batch_query.py` 在**單一瀏覽器 session** 中連續查詢所有問題。腳本會自動將 `references/kb-prompt.md` 前置於每個問題。

### 步驟

1. **產生 queries.json**：根據階段 3.5 的「己方推演假說」「對立方反駁假說」「知識缺口清單」，產生查詢輸入檔。每個問題的 `text` 應包含查詢優化策略（見下方），但**不需要**手動加入 `kb-prompt.md` 前綴。

   ```json
   {
     "notebook_id": "[請填入依據上述規則選定的筆記本 ID 或網址]",
     "questions": [
       {"id": "q1", "text": "請仔細檢索所有來源，在 keywords 或 tags 包含「解僱」「最後手段性」的來源中，關於雇主解僱勞工是否符合最後手段性原則，相關的法規、實務與學說見解為何？請列出所有相關裁判字號。"},
       {"id": "q2", "text": "關於績效獎金是否屬於勞動基準法所定義之工資，相關法規及實務見解為何？"}
     ]
   }
   ```

2. **執行批次查詢**：
   ```bash
   python [draft-pleading skill 目錄]/scripts/run.py batch_query.py --input queries.json --output results.json
   ```

3. **讀取結果**：檢查每個問題的回應狀態與內容。

4. **針對性追問（內部知識庫第二輪）**：若有資訊不足的爭點需要追問：
   - 產生新的 `followup_queries.json`
   - 再次執行 `batch_query.py`
   - 重複直到內部知識庫資訊完整或確認為知識缺口

**若需查詢多個筆記本**：每個筆記本分別產生獨立的 `queries.json`，分別執行。不同筆記本不混在同一批次中。

## 查詢優化策略

撰寫問題時，依據知識庫中檔案的結構特性，採用不同策略：

### 策略 A — 利用 keywords / tags 定位

適用於有 front matter 的檔案。在問題中明確引用相關的關鍵字或法律領域標籤：

```
在 keywords 或 tags 包含「解僱」「最後手段性」的來源中，
關於雇主解僱勞工是否符合最後手段性原則，相關法規及實務見解為何？
```

### 策略 B — 利用目錄結構分段查詢

適用於有目錄的檔案。分兩步進行：
- 第一步（定位）：「這個筆記本中，哪些文件的目錄包含與『資遣費計算』相關的章節？」
- 第二步（深入）：「根據上述文件中關於資遣費計算的章節，具體的法規規定與實務見解為何？」

### 策略 C — 多角度深度檢索

適用於大量資料且無結構化 metadata 的檔案（如「裁判暨函釋彙編」）：
1. **多角度提問**：從不同角度分別查詢（法條依據、實務見解、反面見解），避免單次遺漏
2. **要求窮盡搜尋**：明確指示「請仔細檢索所有來源，列出所有相關裁判字號與函釋文號」
3. **交叉驗證**：對關鍵爭點以不同措辭再查一次

> **注意**：視爭點複雜度，可組合使用策略 A + B + C。

## 4.2 第二層：輔助性外部網路搜尋決策與執行

> [!IMPORTANT]
> **外部搜尋作為擴充工具，必須在 NotebookLM 查詢之後執行，且仍不得先行主導整體論證方向。**
> 外部搜尋旨在補充最新見解與填補缺失，但因可能引入雜訊、SEO 污染與論述偏移，查得資料必須經過 LawThinker 嚴格驗證。使用的最大前提是：**使用者指定的 NotebookLM 筆記本永遠是最優先、最主要、最可靠的來源。**

### 啟動條件與搜尋類型

在完成內部知識庫第一層查詢後，應依據爭點狀況啟動以下搜尋：

| 搜尋類型 | 說明 | 啟動時機 |
|----------|------|----------|
| **實務見解擴充型** | 針對本案關鍵爭點，向官方來源尋找更多、更新或更契合客觀事實的法令、裁判或行政機關函釋。 | NotebookLM 查得見解較少，或需確認是否有最新實務見解時。 |
| **字號驗證型** | NotebookLM 已找到裁判字號、函釋字號，需補上官方或可驗證網址。 | 內部查得具體字號但無原文或網址時。 |
| **知識缺口補強型** | 知識缺口清單 `[待補：...]` 經內部知識庫查詢仍未補齊。 | 當內部知識庫確定無法填補某項缺口時。 |
| **次級文獻萃取型** | 需要補充專業次級文獻的論證語境，但必須能回扣到具體字號或法規。 | 需要強化論理深度，且準備從中萃取具體字號時。 |

### 允許用途

- ✅ 在內部查詢奠基後，擴充更多有助於本案的具體法令、裁判字號、函釋。
- ✅ 驗證 NotebookLM 找到的裁判字號是否確實存在及取得官方全文。
- ✅ 補齊內部知識庫未能覆蓋的知識缺口。
- ✅ 從專業法律文獻（Tier 2）萃取具體法律依據（字號＋法規條文）。

### 禁止用途

- ❌ **絕對禁止**在尚未查詢 NotebookLM 之前，就先用外部搜尋定調論點。
- ❌ 不得拿一般法律部落格結論直接當論證依據。
- ❌ 不得僅憑未載字號的文章就寫入裁判見解。
- ❌ 不得因搜尋到不同立場文章而直接改寫既有攻防主軸（除非已於驗證報告中明確揭示並經使用者確認）。
- ❌ 不得將外部搜尋結果與內部知識庫結果不做區分地混在一起，所有外部來源必須進入「LawThinker 驗證框架」接受嚴格驗證。

### 搜尋策略與關鍵詞模板

#### 實務見解擴充型搜尋

用途：擴充尋找官方的高質量實務見解（裁判、法令、函釋），以補強或豐富論證。

```
搜尋模板："{法律爭點關鍵詞}" 判決 site:judicial.gov.tw
替代模板："{法律爭點關鍵詞}" "{相關法條}" 函釋 site:moj.gov.tw
```

- **強制限定網域**：強烈建議加上 `site:judicial.gov.tw`、`site:moj.gov.tw` 或 `site:mol.gov.tw`，直接從官方獲取 Tier 1 資料。
- 將查獲的新字號送入驗證報告，以 LawThinker 框架檢驗。

#### 字號驗證型搜尋

用途：確認已知字號的正確性與取得官方連結。

```
搜尋模板：site:judgment.judicial.gov.tw "最高法院 1XX 年度台上字第 XXX 號"
替代模板："{完整裁判字號}" 判決 要旨
```

- 優先搜尋官方網域（`judgment.judicial.gov.tw`、`law.moj.gov.tw`）
- 可加網域限制：`site:judicial.gov.tw` 或 `site:moj.gov.tw`

#### 知識缺口補強型搜尋

用途：針對內部知識庫查無的特定法律爭點，定向補齊。

```
搜尋模板："{法律爭點關鍵詞}" "{相關法條}" 判決字號 site:judicial.gov.tw
替代模板："{法律爭點}" 實務見解 裁判 "{法條名稱}"
```

- 優先官方來源與專業法律資料庫
- 搜尋結果必須能回扣到具體裁判字號或法規條文

#### 次級文獻萃取型搜尋

用途：從學者論文、律師專欄等萃取可用的法律依據。

```
搜尋模板："{法律爭點}" 學說 論文 期刊 "{相關法條}"
替代模板："{法律爭點}" 律師 評析 判決 見解
```

- 搜尋結果中必須能萃取出具體裁判字號／函釋字號／法規依據
- 若搜尋結果無法回扣具體法律依據，應標記為「僅背景參考」
- 不得僅抄評論或結論

> [!WARNING]
> **搜尋結果分級處理**：
> - 搜尋到官方來源（`judicial.gov.tw`、`law.moj.gov.tw`、`mol.gov.tw`）→ Tier 1
> - 搜尋到律師/學者/法學資料庫 → Tier 2（須萃取具體法律依據）
> - 搜尋到一般網頁/部落格/論壇/新聞 → Tier 3（僅背景參考，不得入草稿論證）

### 搜尋結果結構化輸出格式

每筆外部搜尋結果必須依下列格式整理，供後續驗證報告使用：

```
## 外部搜尋結果 #[編號]

- **來源層級**：Tier [1/2/3]
- **網址**：[URL]
- **擷取裁判字號／函釋字號**：[字號，或「無」]
- **擷取法規條文**：[法條，或「無」]
- **核心見解**：[一段摘要]
- **是否僅屬背景參考**：[是/否]
- **是否可進入驗證報告候選**：[是/否]
- **風險註記**：[例如：僅評論、無法定位原始裁判、內容可能片面、來源疑似 SEO 文章]
```

### 外部搜尋 Fallback 規則（遇阻時的退路）

> [!WARNING]
> 外部搜尋並非必要步驟，**不得因外部搜尋失敗而阻塞整體流程**。遇到以下情況時，依對應退路處理：

| 失敗情境 | 判斷標準 | 退路規則 |
|---------|---------|---------|
| **遭防火牆/防爬蟲阻擋** | 收到 403、429、Cloudflare 挑戰頁、或 CAPTCHA | 立即停止對該網域的嘗試，改對同一爭點換用其他合規關鍵字或其他官方網域（如 `judicial.gov.tw` ↔ `moj.gov.tw` ↔ `mol.gov.tw`）。若重試仍失敗，標記「外部搜尋阻擋，知識缺口留待案件進行中補充」，直接進入驗證報告。 |
| **搜尋逾時或無回應** | 搜尋工具逾時、無結果頁面 | 同上，不得無限等待或重試超過 2 次。標記「搜尋超時」後進入驗證報告。 |
| **搜尋有結果但全屬 Tier 3** | 所有結果均為部落格、論壇、新聞；無法萃取具體字號或法條 | 不得強行使用，直接在驗證報告的「外部驗證與補漏結果」區塊記載「本次搜尋（[爭點]）僅得 Tier 3 結果，不得入草稿」，繼續以 NotebookLM 結果為主。 |
| **NotebookLM 主查詢已充分** | NotebookLM 已對所有爭點提供 Tier 1 字號與要旨 | 外部搜尋可縮短為「字號驗證型」，僅確認官方連結，不需全面啟動四種搜尋類型。 |

## 4.3 更新撰寫計畫（需使用者確認）

所有查詢完畢後，**絕對不可直接生成草稿**。必須先完成驗證報告（見 `lawthinker-verify.md`），等待使用者確認後，再根據查詢結果更新階段 3 的撰寫計畫，向使用者呈現修訂版：

- 原有論點是否因查詢結果而需調整方向？
- 知識庫中發現的重要判例、法規是否補強或改變了原有論點？
- 是否有新的論點值得增加？
- 各論點預計引用的法律依據（法規與條文、裁判及其字號、行政機關函釋及其文號、學者見解及其來源）
- **外部搜尋材料**：僅將通過驗證報告且經使用者確認的材料納入撰寫計畫

> [!IMPORTANT]
> 更新後的撰寫計畫仍必須嚴格遵守層級順序：`一、` → `(一)` → `1.` → `(1)`。

**必須等使用者確認修訂後的撰寫計畫，才能進入階段 5。**
