## ⛔ 草稿產出前強制預檢（禁止跳過）

> [!CAUTION]
> **每次產出任何草稿文字之前，AI 必須先在對話框輸出以下格式的實體核對清單，使用者看到全部打勾後，才能開始書寫草稿正文。跳過任何一項均屬違規。**
>
> 這份預檢清單設計為「強制輸出」而非「內部推理」，因為書狀格式錯誤（漏寫狀首、地址放錯行、狀尾日期順序顛倒）是最常見且最嚴重的失誤，唯有讓律師親眼確認，才能防止遺漏。

**書寫草稿前，務必先輸出以下清單（不得省略）：**

```markdown
📋 草稿預檢 — 全部確認後才開始書寫

- [x] 案件資料：已讀取案由、案號/股別、法院名稱、當事人稱謂/姓名/地址/電話/電子郵件信箱
- [x] 狀首格式：書狀名稱（H1）、狀首日期（依偏好設定）、訴訟標的金額或價額（依書狀類型）均已確認
- [x] 當事人行：稱謂+姓名+地址寫在同一行；同稱謂多人以頓號合併，不手動換行
- [x] 狀尾順序：謹狀 → 公鑒 → 日期（中華民國…）→ 具狀人 → 法定代理人（若有）→ 訴訟代理人（若有）
- [x] 無審級字眼：書狀名稱不含「二審」「三審」「更一審」等（除非使用者明示）
- [x] 一級標題品質：逐一確認所有 `一、` 標題均為實質結論句，無「於法有據」「不足採信」「無理由」「顯無理由」等空洞修飾語
- [x] 聲明/理由連動：本書狀【有／無】聲明段落 → 對應【必須寫／絕對禁止寫】「理由」或「事實與理由」標題（已確認）
```

> [!NOTE]
> 上述「狀首日期偏好設定」與「案件基本資料讀取」的完整邏輯，詳見 SKILL.md 的「前置動作 0 / A / B」。本預檢清單不重複，僅確認已完成。

> [!IMPORTANT]
> **格式要點（所有書狀通用）**：
> ① 書狀名稱（H1）在第一行；
> ② 狀首日期緊接第二行，**不加任何紀元**（如 `115年4月18日`）；
> ③ 當事人行：稱謂+姓名+地址，**必須寫在同一行**，以空白分隔。地址前綴（引導詞）僅適用於狀首當事人區域：**自然人使用「住」**、**公司或法人機關使用「設」**。若同稱謂有多人且地址相同（或同上、詳卷等），姓名必須以**「頓號」（、）合併相連**（例：`法定代理人 甲〇〇、乙〇〇 住同上`），排版引擎會自動處理每行 26 字之 3 公分定位點折行，AI **絕對禁止**手動為當事人換行；**訴訟代理人（律師）的聯絡地址一律使用「事務所：」標籤**，不使用「住」，格式為：`訴訟代理人 陳〇〇律師 事務所：臺北市〇〇區〇〇路〇段00號 電話：00-00000000`；複數律師共用同一事務所時：`訴訟代理人 陳〇〇律師、林〇〇律師 事務所：臺北市〇〇區〇〇路〇段00號 電話：00-00000000`；> [!CAUTION]
> 上述範例中的「〇」及「00」為**說明用佔位符**，僅用於示意格式。撰寫正式書狀草稿時，**絕對禁止**將「〇」或任何佔位符寫入當事人姓名、地址或聯絡資訊欄位；一律填入使用者實際提供的真實資料。
> ④ 狀尾日期使用「**中華民國**」完整前綴，排在「公鑒」行之後、「具狀人」之前，兩者之間**空一行**；
> ⑤ 聲明與理由區塊大標題前加 **2 個全形空白**，且**必須加粗**。理由區塊大標題**僅能生成「事實與理由」或「理由」**，絕對禁止生成「答辯理由」、「上訴理由」等其他名稱（例：`**　　事實與理由**`）；
> ⑥ 準備書狀、答辯狀、上訴理由狀（無追加/反訴時）**不寫「訴訟標的金額或價額」行**。

## 草稿產出形式

在工作目錄（workspace）中建立 Markdown 檔案，將書狀內文完整寫入。

- **草稿第一行強制要求**：必須是書狀名稱，使用 H1 格式（如 `# 民事起訴狀`）。**絕對禁止**在書狀名稱中加入「二審」「三審」「更一審」等審級字眼（除非使用者在指令中明確特別要求）。即使工作區內既有的舊書狀標題包含了審級字眼，AI 也**絕對不可盲目仿效**。
- **檔名生成規則**：保存草稿前，先列出工作目錄中現有的書狀檔案（`.md` 為主），分析其命名規則（如「案號＋書狀名稱」、「日期＋書狀名稱」等），依同一規則為本次草稿命名。若工作目錄無既有書狀可供參照，退回預設格式：`書狀名稱_草稿.md`。
- **確認無誤**：**必須等使用者明確表示內文完全正確**後，才能進入階段 6。

## 核心撰寫原則

### 0. 嚴禁在論述中使用表格

書狀正文論述中，一律**絕對禁用 Markdown 表格**。不論計算明細、比較分析或任何其他內容，一律使用層級條列（`1.`、`(1)` 等）表述。除非使用者明確要求，方可例外。

### 0.5 全域字元間距強制規則（禁用盤古之白）

為配合 Word/ODT 底層的「亞洲語言字體排版」自動微調引擎，**絕對禁止**在「中文字」與「阿拉伯數字 / 英文字母」的交界處插入實體的半形空白。
- ⭕ **正確示範**：`115年3月25日`、`甲證1`、`新臺幣55,000元`、`最高法院112年度`。
- ❌ **錯誤示範**：`115 年 3 月 25 日`、`甲證 1`、`新臺幣 55,000 元`、`最高法院 112 年度`。
- **例外保障**：「連續英文單字」之間的半形空白（如 `Business Ethics Code`）為語法所需，**必須保留**。

### 1. 結論先行（CRAC）

`一、` 與 `(一)` 的標題必須是「實質主張」的完整句子：
- ✗ `一、關於解僱合法性`
- ✓ `一、依證人證述，被告公司解僱原告已經符合最後手段性原則`

> [!CAUTION]
> **禁止空洞修飾性評價**：`一、` 與 `(一)` 的標題**嚴禁**附帶「於法有據」「不足採信」「無理由」「顯無理由」等空洞修飾語——這些用語不增加任何實質法律資訊，只是給論點貼標籤。
> - ✗ `二、被告應返還借款本金50萬元及約定利息，於法有據`
> - ✓ `二、被告應返還借款本金50萬元及約定利息`
> - ✗ `三、被告所辯不足採信`
> - ✓ `三、被告並未返還借款`（應視被告具體抗辯內容，以結論直接反駁）
>
> **允許實質法律結論**：若用語本身構成有內容的法律判斷（如「不合法」「違反最後手段性原則」），且該論點是為後續同級論點（如請求給付項目）做鋪墊，則允許使用。但具體的論證理由應放在次級層級展開，不堆疊於同一級標題。
> - ✓ `一、被告以虧損為由資遣原告，不合法`（實質法律結論，為後續同級論點鋪墊）
>   - `(一) 被告主張之虧損金額與財報數據矛盾`（論證理由放次級）

### 2. 層級順序（嚴格遵守）

| 層級 | 格式 |
|------|------|
| 第一層 | `一、` |
| 第二層 | `(一)` |
| 第三層 | `1.` |
| 第四層 | `(1)` |

**強制規則：**
- **括號全半形規定**：第二層級 `(一)` 或 `（一）`，排版引擎同時兼容半形與全形。第四層級阿拉伯數字括號維持半形 `(1)`、`(2)`。
- **禁止跳層**：`一、` 之下必須先用 `(一)` 展開；`(一)` 之下必須先用 `1.` 展開，依此類推。
- **程序事項**：起訴狀「事實與理由」中的「程序事項」套用 `一、程序事項`，其餘論點依序遞增。

### 3. 高精確度證據引用

> [!IMPORTANT]
> **排版引擎（`check_citations.py`）已自動攔截以下六類引用格式違規並阻斷排版：裸露證據編號、書證/筆錄缺原文引號、裁判缺引號、裁判疑似改寫、缺頁碼佔位符。AI 只需專注於「如何寫出高品質的引用」，格式稽核由系統自動保障。**

- **引用目標**：關鍵證詞、法規或契約條款必須使用 `「……」` 摘錄原文，引文後以括號注明出處。
- **卷宗稱謂**：遞交給「目前審理之法院」的書狀，提及該法院卷宗時**絕對必須使用「鈞院卷」**，嚴禁「本院卷」。目前為二審時，提到原審卷宗一律稱「一審卷」或「原審卷」。
- **近期筆錄引用**：引用尚無卷宗頁碼之近期筆錄，直接以「日期＋筆錄＋頁次」特定（例：`請見115年3月25日準備程序筆錄第3頁`）。
- **頁碼佔位符格式**：無法確認頁碼時，一律使用 `第 [待補] 頁`。
- **己方新證據豁免**：本次首度提出之新證據，**絕對禁止**加上卷宗頁碼描述。
- **既有證據智動檢索**：引用同審級既有證據前，AI 須先搜尋工作區內既有書狀（**優先檢索 `.md` 檔**，無則檢索 docx/odt），有頁碼則帶入，無則僅列編號。
- **證據射程邏輯規則**：引用證據時，「此有……為憑」或「……可證」收尾句，必須緊接在該證據**實際能證明的事實**之後。嚴禁將證據引用安插在該證據無法證明的事實（如消極事實「未清償」「未給付」）之後。AI 在安插證據引用前，須自問：「這份證據能證明前面這句話嗎？」若不能，則應將引用位置前移至該證據確實能佐證的事實段落之後。
  - ✗ 錯誤：`雙方約定清償期為113年12月31日。屆期後被告未清償本金，此有借據（甲證1）為憑。`（借據無法證明「未清償」）
  - ✓ 正確：`雙方約定清償期為113年12月31日，此有借據（甲證1）為憑。詎料屆期後被告未清償本金。`
- **證據引用句式強制規則**：正文中引用證據時，括號前須說明證據名稱或性質，並以「為憑」或「可證」收尾。
  - ✗ 錯誤：`被告預告終止契約（甲證3）。`
  - ✓ 正確：`被告預告終止契約，此有113年4月2日資遣預告電郵及通知書（甲證3）為憑。`
- **裁判引用強制規則**：引用法院裁判或行政機關函釋時，**必須同時提供**：
  1. **完整字號**（例：「最高法院 112 年度台上字第 584 號民事裁定」）
  2. **裁判或函釋原文逐字引用**，以 `「……」` 引號摘錄，**嚴禁**以 LLM 改寫或概述大意替代原文
  3. **允許省略**：省略中間不影響引用核心意旨的部分，省略處以 `……` 表示
  4. **來源只有大意時**：改用以下格式：`按最高法院 XX 年度台上字第 OOO 號判決意旨（來源僅載大意，原文待核）：【 [大意描述，無引號] 】。`
  5. 若僅有字號完全無大意，標註 `[待補裁判要旨]`
  6. **法院名稱簡寫**：「臺灣高等法院」裁判簡寫為「高院」；「臺灣高等法院臺中分院」簡寫為「臺中高分院」
- **外部來源使用規則**：僅 Tier 1（官方來源）與 Tier 2（專業次級文獻）且已通過「🔍 法律見解深度驗證報告」驗證並經使用者確認的材料，方可寫入草稿；Tier 3（一般網頁）不得作為正式法律依據。
- **證據編號順序**：各項甲證（乙證、上證等）的編號，**必須嚴格依照在書狀正文中被首次引用的先後順序編排**，不得跳號或倒序。
- **證人詰問逐字稿引用格式（全書狀通用）**：採「三段式高強度引證法」：
  1. **前言標題**：以編號層級（如 `1.`）點出待證事實的主張
  2. **Bullet 標示出處與問答**：第一點標註卷證出處，後續呈現詰問對答 `問：...` 與 `證人：...`
  3. **編號總結**：切回編號層級，以 `綜合前述證人證述，足證...` 建立 CRAC 連結

### 4. 具體化攻擊防禦

- 指責對造違反「經驗法則」或「論理法則」時，必須具體指出違反了什麼
- 駁斥必須直指矛盾核心，不得空泛

### 5. 單一論點的雙軌排版

當某層級下僅有一個獨立論點，不展開次級編號，改用以下雙軌制：

**情境 A — 短論述（接排）**：標題後接冒號，內文直接接排
```
一、被告之抗辯無理由：因雙方未簽訂系爭契約。
```

**情境 B — 長論述（換行縮排）**：標題獨佔一行，內文為無編號的縮排段落
```
一、被告之抗辯無理由
    被上訴人主張原告於任職期間有重大過失......（長段論述）
```

**判斷邏輯**（OR Logic，符合任一條件即為「長論述」）：
1. 內文超過 **80 個中文字**
2. 內文包含 **3 個以上句號**
3. 內文包含法條引用或判決引用
4. 內文本身包含次級列點

**例外（強制接排）**：標題本身即完整句子，且內文僅為一句補充說明。

### 6. 論點標題結構與粗體

> [!NOTE]
> **粗體由底層排版引擎（`build_pleading.py`）自動判定，AI 無需手動控制：**
> - 某編號段落下方有子論述（`1.`、`2.` 或縮排段落）→ 系統強制全段加粗（標題）
> - 某編號段落下方無子論述 → 系統視為「實質內文」，絕對不加粗（即使 AI 手動加 `**` 也會被剝除）
>
> **AI 的責任**：確保論點標題（`一、`、`(一)` 等）簡短扼要，僅呈現核心結論（Assertion），不在標題塞入長篇說理或判決原文。

## 書狀內文結構

### 0. 狀首欄位排版與生成規則

| 書狀類型 | 「案號」「股別」 | 「訴訟標的金額或價額」 |
|---------|--------------|----------------|
| **民事起訴狀**（尚無案號） | **不寫** | **必寫**，置於當事人欄位之前。格式：`訴訟標的[XX]：新臺幣[金額]元` |
| **民事上訴狀**（正式提起上訴） | 正常寫「案號」「股別」 | **必寫**，置於**股別之下** |
| **有追加聲明或提起反訴** | 正常寫「案號」「股別」 | **必寫**，置於「案號」**之上** |
| **準備書狀、答辯狀、上訴理由狀**（案件已繫屬且無追加） | 正常寫 | **通常不寫** |

> [!NOTE]
> **排版引擎處理邏輯（AI 生成 Markdown 時無需手動加空白排版）**：稱謂頂格，稱謂與姓名之間**不要有全形冒號**。排版引擎會動態轉成 3cm / 6cm tab stop 排版並套用 `書狀_狀首當事人`。

### 1. 聲明與標題排版（視需要撰寫）

- 撰寫原告「起訴狀」、被告或被上訴人第一次的「答辯狀」、上訴人「上訴狀」，或使用者特別指示修改聲明時，有需要撰寫「聲明」區塊。
- 有寫「聲明」時，才需要寫出「事實與理由」（或「理由」）的標題。
- **標題格式精確規範**：「聲明」與「理由」等區塊大標題，前方**必須且只能**留有 **2 個全形空白**，且**必須使用粗體**（例如 `**　　訴之聲明**`、`**　　理由**`）。**絕對禁止**使用 HTML 註解標示區塊標題。
- **原告起訴狀聲明規範**：
  - **假執行之宣告**：勞動訴訟原告無需撰寫；一般民事訴訟原告必須撰寫（「請准原告提供擔保，宣告假執行。」）
  - **訴訟費用負擔**：無需撰寫
- **被告答辯狀聲明規範**：勞動訴訟被告，或原告有假執行聲明的一般民事訴訟，需加入：「如受不利判決，請准被告提供擔保，免予假執行。」

### 2. 內文推演邏輯（CRAC，起訴狀為例）

- 起訴狀的「事實與理由」第一段，若有「管轄」或「請無庸經強制調解程序」（僅勞動訴訟原告），必須先撰寫。
- **一、本件事實經過**（預設標題）：依時間序列鋪陳案件事實（人、事、時、地、物）。**絕對禁止**使用「不含法律評價之客觀事實」等教條式字眼。內容維持客觀，禁止偷渡法律要件評價或結論。
- 事實段落之後，進入法律論述：
  - **(一) 指摘違法事實（引出請求權基礎）**
  - **(二) 結論式主張（CRAC 結構之 C）**
  - **1. 請求權基礎之確立 (Rule - "按...")**：明確指出法條及內容，拆解為「構成要件」與「法律效果」。實務見解（裁判、行政函釋）、法律條文、學說見解全部屬於 R 層，應在 R 的主幹段落中引述，或以子層級（如 `(1)`）逐一列出。**嚴禁**將裁判見解放在 A（涵攝）或 C（結論）層級下。
    - 正確結構範例：
      ```
      1. 按勞基法第11條第2款…（法律條文）
      (1) 最高法院107年度台上字第274號判決指出：「…」（裁判見解）
      (2) 臺灣高等法院107年度…亦指出：「…」（裁判見解）
      2. 查本件…（A：涵攝事實）
      3. 是以…（C：結論）
      ```
  - **2. 構成要件之涵攝 (Application - "查本件...")**：將客觀事實套入構成要件檢驗。若涉及金錢請求，必須以層級條列（`1.`、`2.`…）列各項請求的期間、計算式與金額，最後加入小計，**絕對不得使用表格**。
  - **3. 結論 (Conclusion - "是以...")**：總結涵攝結果。

### 3. 一般書狀理由撰寫格式

從 `一、` 開始，每個爭點一個 `一、` 級標題。第一層必須是「實質主張」的結論式完整句子。若不需要聲明，直接從 `一、` 開始撰寫理由。

> [!CAUTION]
> **【最易復發的兩個格式錯誤——每次產出草稿前必查】**
>
> **① 一級論點標題（`一、`）品質**：標題必須是有實質內容的法律結論句。**嚴禁**附帶「於法有據」「不足採信」「無理由」「顯無理由」等空洞修飾語，這些詞語不傳達任何具體法律主張，只是貼標籤。
> - ✗ `三、被上訴人「民事答辯三狀」之主張顯無理由，上訴人綜合反駁如下。`
> - ✓ `三、被上訴人持續隱匿親屬身分主導異常過帳已構成結構性背信，且高層從未授權此利益衝突行為。`
>
> **② 聲明/理由標題連動**（上訴理由二狀以後、準備書狀、答辯狀後續等**最易犯錯**）：
> - **有聲明段落** → **必須寫**「事實與理由」或「理由」標題
> - **無聲明段落** → **絕對禁止**出現任何「理由」標題，直接從 `一、` 開始
>
> 補充書狀（上訴理由（二）狀以後、準備書（二）狀以後等）因通常**省略聲明**，故在開頭句「為就……事：」之後，**絕對不應出現** `**　　理由**` 的過渡標題。

## 特殊書狀格式：聲請調查證據

若書狀包含（或專門為）聲請調查證據，依以下格式撰寫：

1. **標題**：使用 `一、` 或 `二、` 等層級標題，例如 `一、聲請傳訊證人〇〇〇`
2. **證人地址**：接續在證人姓名之後以全形括號標註
   - 有地址：`一、聲請傳訊證人〇〇〇（地址：〇市〇區〇〇路〇號）`
   - 無地址：`一、聲請傳訊證人〇〇〇（地址：[請填寫證人地址]）`
3. **待證事實與理由**：使用 `(一) 待證事實：` 與 `(二) 聲請理由：`

**調查對象地址填寫規則**：
- 若向機關、法人或公司請求函查，應使用網路搜尋查詢其**實際地址（含郵遞區號）**後填入。
- 若調查事項屬於下列**免填清單**，則調查對象與對象地址欄位**一律留空不填**（法院可自電子閘門或 WebIR 系統調取）：
  - 戶籍資料、勞工保險投保資料明細表、勞工退休金個人專戶明細與提繳紀錄
  - 財產歸屬資料清單（不動產、車輛、投資等）
  - 綜合所得稅各類所得清單、健保投保歷史資料、車籍資料
  - 土地建物登記謄本資料、地籍與測量圖資、異動索引、地價資料
  - 臺灣高等法院被告前案紀錄表

## 證據或文件清單區塊（位於「謹狀」之前）

- **位置**：正文結束後空一行，再接「證據或文件清單：」；區塊結束後空一行，再接「謹狀」。
- **【重要：無證據時之處理】若整份草稿無任何證據及附件引用，【絕對不得】產生「證據或文件清單：」及其後續證據清單。**
- **格式與段落樣式**：
  - 第一行：「證據或文件清單：」（無縮排）。
  - **【模式 1：純文字清單（預設）】**：證據編號內容懸掛縮排（如 `甲證1：...`，使用純文字）。
  - **【模式 2：三欄表格（經使用者明確要求時啟動）】**：欄位由左至右必須精確為「編號」、「名稱或內容」、「頁碼」。
- **前綴選擇**：
  - 一審原告/聲請人：`甲證` / `原證`；一審被告/相對人：`乙證` / `被證`。循既有書狀習慣沿用。
  - **二審之甲證/乙證體系**：一審使用「甲證/乙證」，二審原則上**續接編號**（如：一審最後為甲證5，二審第一件新證據編為甲證6）。
  - **二審之原證/被證體系（強制轉換）**：二審**絕對不可**繼續使用「原證/被證」。必須視當事人身分強制轉換：若為「上訴人」，改從「上證1」開始；若為「被上訴人」，改從「被上證1」開始。
- **編號延續**：偵測既有書狀最後編號，從下一號開始。
- **附件**：與證據採獨立編號，前綴為 `甲附件`、`乙附件` 等。

> [!IMPORTANT]
> **【與高院證據清單表的互斥關係】**
> 「證據或文件清單：」區塊是通用格式，適用於所有書狀。
> 若使用者要求的是**高院格式的 7 欄證據清單表**（含「證據附卷位置」合併標頭等），應交由 `pleading-table` 技能處理，**不在本技能的 Markdown 草稿中產出**。

> [!IMPORTANT]
> **【附件連動規則：高院證據清單表】**
> 當使用者表示書狀需附上高院証據清單表時，「證據或文件清單：」區塊末尾，**必須加上**以下一行：
>
> ```
> 附件：證據清單
> ```
>
> 此行**不佔用**證據編號，以「附件」前綴獨立處理。

## 狀尾排版規範（同頁不分割）

包含「謹狀」、「[法院名稱]民事庭 公鑒」、「日期」、「具狀人」及「訴訟代理人」。

> [!IMPORTANT]
> **【日期排序強制規則】**：「日期」行必須排在「公鑒」行之後、「具狀人」之前。
>
> 正確順序：`謹狀` → `[法院]公鑒` → `日期` → `具狀人` → `法定代理人`（若有）→ `訴訟代理人`（若有）

**狀尾關鍵規則**：
- 「公鑒」前留**一個全形空白**，法院全名必須緊湊，不含任何多餘空格（例：`臺灣臺北地方法院民事庭　公鑒`）
- 日期行使用**「中華民國」**完整前綴（如：`中華民國115年3月10日`），**絕對禁止**加「日期：」等任何前綴詞
- 「具狀人：」「訴訟代理人：」各行職稱之後必須使用**「全形冒號」**

**【日期生成規則】**：
- **若使用者在指令中未明確要求指定日期**，一律預設為**撰寫當日之後第2日**（遇例假日順延至次一工作日/星期一）。
- AI 必須自行計算確切「年、月、日」，**禁止**僅留空白占位符。
- **修改補充模式之強制覆寫**：在後續「修改」、「補充」或「改寫」既有草稿時，**絕對必須重新計算並覆寫**草稿內的狀首與狀尾日期，嚴禁照抄舊草稿的過期日期。
- **【全域年份紀元強制規則】**：所有地方的年份預設不加紀元，但有兩處例外：（1）「聲明」中加「民國」；（2）「狀尾日期」（公鑒之後、具狀人之前）加「**中華民國**」完整前綴。其餘所有地方**絕對不加**任何紀元前綴。
