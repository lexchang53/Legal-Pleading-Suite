# 爭點整理狀附表格：完整操作手冊

> [!IMPORTANT]
> **本文件只適用於使用者明確要求「附表格爭點整理狀」時。**
>
> | 使用者需求 | 草稿格式 | 排版交接 |
> |-----------|---------|---------|
> | 撰寫**附表格爭點整理狀** | 雙格式：人讀清單 + 下半部 JSON | → `pleading-table` |
> | 撰寫**一般爭點整理狀**（未要求附表格） | 一般 Markdown 書狀 | → `內部排版引擎` |
>
> 若無附表格需求，不使用本文件。

> [!CAUTION]
> 其他書狀類型，包括準備書、答辯狀、上訴理由狀、聲請狀等，即使內容涉及調查證據，也**不得**使用本文件中的「聲請調查證據表」流程。

> [!NOTE]
> 本流程已改為：
> - `draft-pleading` 產出雙格式草稿
> - `pleading-table` 讀取最後一個 JSON 區塊
> - 以 `python-docx` + `table-tmpl.docx` 直接生成 DOCX
>
> 舊的 `docxtpl`、XML merge、`generate_pleading.py`、`merge_table.py` 等流程均不再使用。

---

## 一、草稿格式總原則

同一份 `.md` 草稿中，必須同時包含兩部分：

1. 上半部：人可閱讀與修改的清單草稿
2. 下半部：供 builder 使用的完整 JSON

> [!IMPORTANT]
> 上半部清單與下半部 JSON 必須一一對應。
> 若使用者修正清單內容，JSON 也必須同步更新。

> [!IMPORTANT]
> **草稿生成後、呈現使用者前的自動比對規則**：
> 在附表格爭點整理狀草稿完成後，AI 必須先**自動比對**上半部人讀清單與下半部 JSON 的內容：
> - `答辯原因事實` ↔ `reasons`
> - `法律依據` ↔ `laws`
> - `證據` ↔ `evidences`
> - `聲請調查證據表` ↔ `evidence_request.items`
>
> 若發現不一致，**一律以上半部人讀文字為準，自動修正 JSON**，再呈現最終草稿。

> [!CAUTION]
> `pleading-table/scripts/extract_issue_json.py` 只會提取最後一個 ` ```json ... ``` ` 區塊。
> 不得依賴解析上半部 Markdown 來補資料。

## 二、草稿上半部格式

### 基本結構

上半部清單至少應包含：

1. 聲明
2. 不爭執事項
3. `{party_status}爭點整理表`
4. 聲請調查證據表（若適用）
5. 表格後補充論述（若有）

最終 DOCX 主體段落順序為：

1. `一、聲明：`
2. `二、不爭執事項：`
3. `三、{party_status}爭點整理表`
4. `四、聲請調查證據表`（僅在 `evidence_request.items` 非空時輸出）

### 聲明清單格式

- **【絕對強制】聲明還原與訴訟費用排除規則**：記載方式應如當事人上訴狀或答辯狀的原始聲明記載，原來有幾項就列幾項，**絕對不要**強制將2項以下的聲明合併為1項。
- **【絕對強制】**關於「訴訟費用負擔」的聲明，在爭點整理狀中**絕對不要寫**。

#### 模式 A：單項聲明

若原始聲明內容本就只有一項，可直接寫成：

```markdown
1. 聲明：確認兩造間僱傭關係存在。
```

#### 模式 B：多項聲明

若原始聲明為多項（例如原判決廢棄、駁回一審之訴等），請依原樣分項列出（但必須排除涉及訴訟費用負擔的項目），對應最終 DOCX 的 `通用_層級2`：

```markdown
1. 聲明：
- （一）原判決廢棄。
- （二）廢棄部分，被上訴人在第一審之訴駁回。
```

> [!IMPORTANT]
> 聲明**不是**一律不得分項。
> 是否分項，取決於請求內容是否為多個獨立聲明事項。

### 不爭執事項清單格式

不爭執事項亦有兩種合法模式：

#### 模式 A：單項引用式

**【絕對強制】**若原審（一審）判決已有整理「不爭執事項」，且當事人未特別表示相反意見者，必須使用此模式單項引用，不可逐項列出。
**引用文字必須完全依照一審判決書該段落的標題層級與用語：**
例如，若一審判決書是在「事實及理由」下的「貳、實體方面」下的「三、兩造不爭執事項：（或兩造不爭執之事項：）(本院卷第XXX頁...)」，則必須對應標籤出判決書之層次結構，記載為：`如原審判決書「事實與理由」第貳、三項所列「兩造不爭執事項」各項所載。`
（**注意**：引用標題時，原來標題後面的冒號、括號，以及括號內的文字，都**絕對不要**加上去。）

```markdown
2. 不爭執事項：如原審判決書「事實與理由」第貳、三項所列「兩造不爭執事項」各項所載。
```

#### 模式 B：多項列舉式

若原審判決**沒有**整理「不爭執事項」，則檢查雙方書狀內容，若有主張相同事實者，即為「不爭執事項」；或是原審判決雖然有整理「不爭執事項」，但當事人**不同意**其中一部分，並**特別列出來**，則應連同其他**沒有**表示**不同意**的「不爭執事項」，逐項全部列出。對應最終 DOCX 的 `通用_層級2`：

```markdown
2. 不爭執事項：
- （一）兩造於某年月日簽訂系爭契約。
- （二）上訴人曾於某年月日寄發通知。
- （三）某份書面確由被上訴人作成。
```

### 爭點整理表標題格式【絕對強制】

標題中的 `{party_status}` 應依當事人在一審及二審（或原審與再審）的身分或稱謂**完整併列填入**。
若當事人為「二審上訴人」且為「一審被告」，則 `{party_status}` 必須是「上訴人（即原審被告）」，標題行應呈現為：
`3. 上訴人（即原審被告）爭點整理表`
此規則在下半部的 JSON 中亦同，`party_status` 必須填入完整併列的稱謂，不可僅填「上訴人」。

### 爭點整理表清單

- 爭點整理表在草稿中，以清單方式呈現。  
- 最終 DOCX 中，所有爭點會依照順序合併為單一表格。
- **排列順序【絕對強制】**：應按判斷各項爭點的「法院審理優先順序」排列。例如：首先是「是否有業務緊縮」，其次是「雇主終止契約是否符合最後手段原則」，最後是「若終止合法，應給付多少報酬」。
- 關於「事實上爭點」，「法律依據」可記載「事實認定，依證據判斷」；若涉及舉證責任，應由主張有利事實之對方當事人舉證，則「法律依據」為「民事訴訟法第277條」。
- **法院名稱簡寫【絕對強制】**：「法律依據」中若引用「臺灣高等法院」裁判，應簡寫為「高院」；若引用「臺灣高等法院臺中分院」裁判，應簡寫為「臺中高分院」。
- 「法律依據」部分，若係以法院裁判為依據，**絕對強制：應先列出法院裁判要旨文字，再以括號列出裁判字號**（裁判字號可依上述規則簡略）。
	- 格式：`[要旨]（[裁判字號]）`
	- 例如：`經濟性解僱應重減少人力需求與最後手段性，不拘泥法條文字（最高法院111台上8）`
- 「證據」部分，如果有待調查，可留空白。

範例：

```markdown
3. 上訴人（即原審被告）爭點整理表
- 爭點一：兩造間是否成立僱傭契約
- 答辯原因事實：
	- 被上訴人所稱到職日期與實際情形不符
	- 雙方並未就長期雇用條件達成合意
- 法律依據：
	- 事實認定，依證據判斷
- 證據：
	- 匯款紀錄（某卷某頁、乙證1）
	- 通訊紀錄（某卷某頁、乙證2）

- 爭點二：上訴人依勞動基準法第11條第2款終止契約是否合法
- 答辯原因事實：
	- 上訴人有業務緊縮
	- 上訴人其他部門沒有人力需求
- 法律依據：
	- 勞動基準法第11條第2款
	- 經濟性解僱應重減少人力需求與最後手段性，不拘泥法條文字（最高法院111台上8）
- 證據：
	- 通知函（一審卷第 [待補] 頁、乙證3）
```

### 聲請調查證據表清單

若屬**附表格爭點整理狀**，且案情中有需一併提出的證據調查事項，草稿上半部必須同時提供「聲請調查證據表」的清單版本。  
這一部分是供使用者校對內容，不是最終 DOCX 表格版面。

**調查對象地址填寫規則**：

- 若「調查對象」為政府機關、法人團體或私人公司，**應使用網路搜尋查詢其實際地址（含郵遞區號）後填入**「對象地址及聯絡方式」欄位。
- 若調查事項屬於下列「**免填清單**」，則該列的「調查對象」與「對象地址及聯絡方式」欄位**一律留空不填**（因法院可自電子閘門或 WebIR 系統查詢調取，無需函查外部機關）：
  - 戶籍資料
  - 勞工保險投保資料明細表
  - 勞工退休金個人專戶明細與提繳紀錄
  - 財產歸屬資料清單（查詢登記在名下之不動產、車輛、投資等）
  - 綜合所得稅各類所得清單（查詢薪資、利息、股利等年度報稅資料）
  - 健保投保歷史資料
  - 車籍資料
  - 土地建物登記謄本資料
  - 地籍與測量圖資
  - 異動索引
  - 地價資料
  - 臺灣高等法院被告前案紀錄表

範例：

```markdown
4. 聲請調查證據表

- 第1項
	- 所涉爭點：爭點二
	- 調查事項：函查被上訴人於某段期間之勞保投保資料
	- 調查對象：勞工保險局
	- 對象地址及聯絡方式：臺北市中正區羅斯福路1段4號
	- 待證事實：被上訴人於系爭期間是否另有工作收入

- 第2項
	- 所涉爭點：爭點一、爭點二
	- 調查事項：訊問證人甲
	- 調查對象：證人甲
	- 對象地址及聯絡方式：地址與聯絡方式待補
	- 待證事實：兩造互動經過及系爭終止前後事實
```

> [!CAUTION]
> **自動增列事實上爭點【絕對強制】**：
> 若使用者有撰寫聲請調查證據事項，而證據事項**無法對應**到已經整理出來的既有爭點（即「所涉爭點」欄位若是空白，或無法給出明確爭點），你**必須**依據該項「待證事實」，在「撰寫計畫」與「草稿」中自動增列一個相對應的爭點。
> 增列後，你**必須**在使用 notify_user 與使用者溝通時，特別提醒：「因為第 X 項聲請調查證據無法對應既有爭點，已自動為您增列對應的爭點 X」。

> [!IMPORTANT]
> 草稿中的「聲請調查證據表」也應採**清單方式**呈現。
> 不必在 Markdown 草稿中模擬 Word 表格版面。

> [!IMPORTANT]
> 若本件**沒有**聲請調查證據事項，則：
> - 草稿上半部不寫第 4 部分
> - 下半部 JSON 不提供 `evidence_request`
> - 最終 DOCX 不輸出「四、聲請調查證據表」及其下表格

## 三、JSON 結構要求

下半部 JSON 必須完整、自洽，且可直接供 `pleading-table-builder` 使用。

根層至少包含：

- `party_status`
- `reason_header`
- `issues`
- `post_table_markdown`（選填）
- `evidence_request`（選填）
- `statement_text` 或 `statement_items`
- `undisputed_text` 或 `undisputed_items`

### 聲明欄位規則

#### 模式 A：單項聲明

```json
{
  "statement_text": "原判決廢棄；廢棄部分，被上訴人在第一審之訴駁回。",
  "statement_items": []
}
```

#### 模式 B：多項聲明

```json
{
  "statement_text": "",
  "statement_items": [
    "原判決關於駁回上訴人後開第二至四項之訴部分暨訴訟費用之裁判，均廢棄。",
    "確認上訴人與被上訴人間僱傭關係存在。",
    "被上訴人應自某年月起，按月給付上訴人新臺幣某元，至復職之前一日止。",
    "被上訴人應自某年月起，按月提繳勞工退休金至上訴人之勞工退休金個人專戶。"
  ]
}
```

### 不爭執事項欄位規則

#### 模式 A：單項引用式

```json
{
  "undisputed_text": "如原審判決書第某頁所列「不爭執事項」各項所載。",
  "undisputed_items": []
}
```

#### 模式 B：多項列舉式

```json
{
  "undisputed_text": "",
  "undisputed_items": [
    "兩造於某年月日簽訂系爭契約。",
    "上訴人曾於某年月日寄發通知。",
    "某份書面確由被上訴人作成。"
  ]
}
```

> [!IMPORTANT]
> `statement_text` 與 `statement_items` 兩者應擇一使用。
> `undisputed_text` 與 `undisputed_items` 兩者亦應擇一使用。
> 不得同時都空，也不得同時都填實質內容。

### 爭點資料結構

每筆爭點物件格式如下（放在 `issues` 陣列中）：

```json
{
  "issue_number": "爭點一",
  "description": "上訴人依勞動基準法第11條第2款終止契約是否合法",
  "reasons": [
    "上訴人有業務緊縮",
    "上訴人其他部門沒有人力需求"
  ],
  "laws": [
    "勞動基準法第11條第2款",
    "經濟性解僱應重減少人力需求與最後手段性，不拘泥法條文字（最高法院111台上8）"
  ],
  "evidences": [
    "通知函（一審卷某頁、乙證3）"
  ]
}
```

### 聲請調查證據表 JSON 結構

僅在**附表格爭點整理狀**且確有此需求時，才可產生 `evidence_request`：

```json
{
  "evidence_request": {
    "items": [
      {
        "related_issues": ["爭點二"],
        "investigation_item": "函查被上訴人於某段期間之勞保投保資料",
        "target": "勞工保險局",
        "target_address_contact": "臺北市中正區羅斯福路1段4號",
        "fact_to_prove": "被上訴人於系爭期間是否另有工作收入"
      }
    ]
  }
}
```

### `evidence_request` 生成時機

應產生 `evidence_request` 的情形：

- 使用者要求撰寫**附表格爭點整理狀**
- 案情中確有需一併呈現的調查證據聲請內容
- 使用者未明示排除該表

不得產生 `evidence_request` 的情形：

- 使用者僅要求一般爭點整理狀
- 使用者要求的是其他書狀
- 本件沒有需列入「聲請調查證據表」的事項

## 四、完整雙格式範例

```markdown
## 爭點整理草稿

1. 聲明：
- （一）原判決關於駁回上訴人後開第二至四項之訴部分暨訴訟費用之裁判，均廢棄。
- （二）確認上訴人與被上訴人間僱傭關係存在。
- （三）被上訴人應自民國114年10月1日起至上訴人復職之日止，按月於次月10日給付上訴人新臺幣（以下同）55,000元，及自各期應給付日之翌日即次月11日起至清償日止，按週年利率百分之五計算之利息。

2. 不爭執事項：
- （一）兩造於某年月日簽訂系爭契約。
- （二）上訴人曾於某年月日寄發通知。

3. 上訴人（即原審被告）爭點整理表

- 爭點一：兩造間是否成立僱傭契約
- 答辯原因事實：
	- 雙方並未就長期雇用條件達成合意
	- 被上訴人所稱到職日期與實際情形不符
- 法律依據：
	- 事實認定，依證據判斷
- 證據：
	- 匯款紀錄（某卷某頁、乙證1）

- 爭點二：上訴人依勞動基準法第11條第2款終止契約是否合法
- 答辯原因事實：
	- 上訴人有業務緊縮
	- 上訴人其他部門沒有人力需求
- 法律依據：
	- 勞動基準法第11條第2款
	- 經濟性解僱應重減少人力需求與最後手段性，不拘泥法條文字（最高法院111台上8）
- 證據：
	- 通知函（某卷某頁、乙證3）

4. 聲請調查證據表
- 第1項
	- 所涉爭點：爭點二
	- 調查事項：函查被上訴人於某段期間之勞保投保資料
	- 調查對象：勞工保險局
	- 對象地址及聯絡方式：臺北市中正區羅斯福路1段4號
	- 待證事實：被上訴人於系爭期間是否另有工作收入

```json
{
  "statement_text": "",
  "statement_items": [
