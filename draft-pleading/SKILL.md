# 訴訟書狀撰寫技能

依據使用者提供的訴訟資料，分析案情、查詢法律知識庫，產出結構嚴謹的 `.md` 法律書狀草稿，並在使用者確認後自動執行自身的腳本 `python scripts/build_pleading.py` 產出最終 Word 檔。

## 觸發條件

- **新撰書狀**：「書狀」「訴狀」「起訴狀」「上訴理由狀」「答辯狀」「準備書狀」「爭點整理狀」、聲請調查證據
- **修改補充**：「補充」「修改」「增加」「新增論點」，以及**任何要求修改訴訟書狀草稿的指令**（包含重新開新對話，或距離第一次形成草稿後幾天提出的修改），皆應主動適用。

> [!NOTE]
> **狀首欄位規則**
> - **民事起訴狀**：不寫「案號」「股別」；「訴訟標的金額或價額」**必寫**。
> - **民事上訴狀**（正式提起上訴）：寫「案號」「股別」；「訴訟標的金額或價額」**必寫**，置於股別之下。
> - **有追加聲明或提起反訴**：寫「案號」「股別」；「訴訟標的金額或價額」**必寫**，置於案號之上。
> - **準備書狀、答辯狀、上訴理由狀**（案件已繫屬且無追加）：正常寫「案號」「股別」；**通常不寫**「訴訟標的金額或價額」。
>
> **【訴訟標的金額或價額　名稱選用規則（依訴之聲明判斷）】**
> - 請求**僅有金錢請求**（給付○元、返還○元、賠償○元等具體金額）：名稱為「**訴訟標的金額**」
> - 請求**僅有非金錢請求**（確認僱傭關係、回復原職、遷讓、拆除、塗銷等非金額給付）：名稱為「**訴訟標的價額**」
> - 請求**混合兩類**（如同時請求確認關係＋給付薪資）：名稱為「**訴訟標的金額及價額**」

## 全域強制規則

> [!IMPORTANT]
> **無干擾與連續執行原則**
> 執行本技能時，必須**全流程連續、自主運作，直到 Markdown 草稿產出並交接完畢**。
>
> **允許中斷的時機**（僅以下節點）：
> - 階段 3：撰寫計畫產出後
> - 階段 4：「🔍 法律見解深度驗證報告」（含防定錨檢核區塊）輸出後，等待使用者確認
> - 階段 5：草稿產出後（使用者確認後自動進入階段 6）
> - 修改補充模式步驟 M3：修改計畫產出後
> - 遇到無法靠推知補齊的致命缺失
>
> **連續執行規則**：
> 1. 所有無破壞性的 `run_command` **務必設 `SafeToAutoRun: true`**
> 2. 將多檔讀取與分析合併在代理內部自主迴圈

> [!CAUTION]
> **絕對禁止**：
> - 捏造不存在的判決字號、法規條文或證據內容
> - 在頁碼不確定時不留占位符（應使用 `第 [待補] 頁`）
> - 將產出檔案存到 `brain`、`artifacts` 或其他系統目錄（必須存到使用者工作目錄）
> - **無證據時產生證據清單：** 若整份草稿無任何證據及附件引用，【絕對不得】產生「證據或文件清單：」及其後續證據列表。
> - **年份紀元強制規則：** 年份預設不加紀元，但有兩處例外：（1）「聲明」中的年份必須加「民國」前綴；（2）「狀尾日期」（公鑒行之後、具狀人之前）必須使用「**中華民國**」完整前綴。其餘地方（狀首日期、理由段落等）年份前**絕對不加**任何紀元。
> - **誤用既有書狀日期：** 即使使用者 `@` 提供含日期的既有書狀，只要未明確指示日期，一律需重新計算預設日期。
> 
> **證據清單生成規則**：
> 「證據欄」用詞全面取代為「證據或文件清單」。預設採用純文字 `甲證1：...` 模式。當使用者明示需要時，才生成 Markdown 三欄表格（編號 | 名稱或內容 | 頁碼）。狀尾日期預設使用「中華民國」完整前綴，生成於「法院公鑒」之下、「具狀人」之上，兩者之間空一行。

**層級順序**（嚴格遵守，貫穿所有階段）：`一、` → `(一)` → `1.` → `(1)`
**單一論點不展開原則**：若某層級下只有單一個子論點，絕對**禁止**產生單一的 `1.` 或 `(1)`。應改用「短論述（接排加冒號）」或「長論述（換行無編號縮排）」。

### 爭點整理表專項強化規則

> 爭點整理表相關強制規則（法律依據格式、證據頁碼、調查對象地址、「提出人」稱謂）全部詳見 `references/phase-c-issue-pleading.md`。



### 來源分級與外部結果隔離原則

> 來源分級（Tier 1/2/3）、外部搜尋結果隔離規則、組標籤型搜尋策略、驗證報告格式，全部詳見 `references/phase4-knowledge-query.md`。

## 工作流程

依工作性質分為**新撰模式**（階段 1～6）與**修改補充模式**（見末尾）。

### 階段 1：接收與解析訴訟資料

收集使用者以 `@` 指定的檔案，並自動掃描工作目錄。

| 格式 | 處理方式 |
|------|----------|
| `.md` | 直接讀取 |
| `.pdf` | 使用 `pdf` skill |
| `.docx` | 使用 `docx` skill 的 `pandoc` |
| `.odt` | `pandoc document.odt -o output.md` |

解析後整理：案件基本資訊、事實經過、爭點、證據清單、聲請調查證據事項、對造主張。

### 階段 2：分析訴訟資料

識別爭點攻防方向 → 擬定結論式主張（`一、` 層級標題）→ 規劃證據鏈與法律依據。

### 階段 3：撰寫計畫（需使用者確認）

**執行前讀取** `references/phase3-writing-plan.md`

**不可遺漏的強制規則**：
1. 起訴狀**必須**包含：聲放、不含法律評價之客觀事實、請求權基礎
2. 勞動訴訟原告**不寫假執行宣告**
3. 論點大綱的層級順序必須與最終書狀一致

**必須等使用者確認後才進入階段 3.5。**

### 階段 3.5：生成推論假說骨架（HyDE 高語境檢索準備）

> [!IMPORTANT]
> **全自動背景執行，不得向使用者確認，不得中斷。**
> 本階段產出物為「內部工作文件」：不單獨呈現給使用者，但須完整傳遞至階段 4，並在階段 4 的驗證報告結尾附上，供使用者事後審閱。

接收階段 3 已確認的撰寫計畫，AI 憑自身預訓練知識自動生成以下三項輸出：

**① 己方推演假說**：支持我方請求權基礎與事實涵攝的完整論證段落（不要求法條項次與字號字面準確，但必須具備完整的法律論證語境，包含構成要件涵攝與法律效果）。

**② 對立方反駁假說**：對造最可能提出的抗辯邏輯或對我方不利的實務見解（以完整段落撰寫，涵蓋對造論點的内在邏輯與可能援引之法律基礎）。

**③ 知識缺口清單**：在上述兩份假說中，對所有缺乏確切法條項次或判決字號之處，以 `[待補：描述所需見解]` 格式標注，並彙整為獨立清單，成為階段 4 的定向查詢驅動來源。

**退路規則**：若因案件資料不足，無法生成有意義的推演假說，須輸出 `[假說生成失敗：資料不足，原因：___]`，並直接以「知識缺口清單」作為唯一查詢驅動來源，進入階段 4。

### 階段 4：查詢知識庫

**執行前讀取** `references/phase4-knowledge-query.md` 與 `references/lawthinker-verify.md`。

**跳過查詢的情況**：
1. 撰寫「爭點整理狀」→ **絕對跳過**（直接進入階段 5）
2. 單純「聲請調查證據狀」且未要求查詢 → **預設跳過**（直接進入階段 5）

#### 四層查詢架構（嚴格依序執行）

**第一層 — NotebookLM / 內部知識庫主查詢**

查詢輸入**不依賴單純關鍵字**，須以「階段 3.5 已生成的雙向假說段落與知識缺口清單」為高語境 Prompt 驅動 NotebookLM：

| 查詢軌道 | 輸入來源 | 查詢目標 | 最低查詢數 |
|----------|----------|----------|------------|
| **己方查詢軌道** | 己方推演假說段落 | 尋找支持或補強我方論點的判決字號、學說、實務見解 | **3 個查詢指令** |
| **對立查詢軌道** | 對立方反駁假說段落 | 尋找對造可能援引的不利見解，及實務上用以區辨（Distinguish）的操作標準 | **2 個查詢指令** |
| **知識缺口補齊** | 缺口清單 `[待補：...]` 項目 | 逐一執行定向查詢，補齊法條項次或判決字號 | 依缺口數量，每項至少 1 個 |

**第二層 — 輔助性外部網路搜尋（擴充與嚴格驗證）**

在完成 NotebookLM 第一層主查詢後，為豐富實務見解並確保資訊為最新，**應主動啟動外部網路搜尋**（特別是針對官方網站如 judicial.gov.tw / moj.gov.tw / mol.gov.tw）。

> [!IMPORTANT]
> - **先後順序**：必須**先**完成內部知識庫（NotebookLM）的主查詢，**後**才進行網路搜尋。使用者指定的 NotebookLM 筆記本永遠是最優先、最主要、最可靠的依據來源。外部搜尋僅作為輔助擴充、補強與驗證使用。
> - **搜尋範圍**：
>   1. **實務見解擴充**：以本案關鍵爭點為關鍵字，搜尋是否有最新或更相關的裁判字號、行政函釋、法規。
>   2. **字號驗證與補漏**：為 NotebookLM 找到的字號尋找官方連結，或補齊知識缺口清單 `[待補：...]`。
> - **LawThinker 嚴格驗證**：所有外部網路查得的資料（法令、裁判、行政機關函釋等），必須強制納入「🔍 法律見解深度驗證報告」，依據知識準確性、事實關聯性與程序合法性進行原子級別的嚴格檢驗，並詳列風險標記。
> - **禁止事項**：不得將未經驗證的外部資料直接混入草稿正文；不得把一般網頁搜尋結果（Tier 3）視為當然可信。

**第三層 — 驗證報告隔離**

在所有查詢完畢後，**絕對不可直接生成草稿**。必須嚴格遵循 `lawthinker-verify.md` 的指示，在對話框輸出「🔍 法律見解深度驗證報告」（含末尾的「📌 防定錨檢核」區塊）。外部搜尋結果必須在報告中獨立標示來源層級（Tier 1/2/3）並附風險標記。

**第四層 — 使用者確認後吸收**

**必須等待使用者確認後，更新撰寫計畫並再次等使用者確認**，才進入階段 5。僅經使用者確認的合格材料，方可吸收入撰寫計畫與後續草稿。

### 階段 5：生成書狀草稿（需使用者確認）

> [!CAUTION]
> **進入本階段前，必須先讀取以下文件，否則將產出格式全面錯誤的草稿：**
> - `references/phase5-draft-rules.md`（含強制預檢清單、四類黃金範例、完整格式規範）
>
> 該文件定義了草稿的精確格式（例如：第一行必須是 H1 書狀名稱、禁止使用 `##`/`###` 標題、聲明/理由標題前要加全形空白、狀尾日期位置等）。不讀取此文件，發生格式錯誤是**必然的結果**。

若撰寫**爭點整理狀**，改為讀取 `references/phase-c-issue-pleading.md`。

**前置動作 A — 讀取案件基本資料（重要）**：在正式生成草稿前，AI 必須依以下順序讀取當事人與法院資訊，以填入狀首及狀尾：
1. 優先搜尋工作目錄下名為「案件基本資料」的檔案（**限 `.md`、`.txt`**），讀取：訴訟當事人稱謂、姓名或名稱、地址、電話、電子郵件信箱、法院名稱、以及**案由**。
2. 若無「案件基本資料」檔案，再尋找工作目錄中**最近日期**的書狀檔案（優先 `.md`，次為 `.odt`、`.docx`），從其狀首中擷取上述資訊。

**「案由」填入規則**：讀取到案由後，將其填入書狀開頭句「為就……事件，……事：」，並依下表確認動詞：

> [!NOTE]
> **序號與動詞關係《絕對強制》**：書狀名稱中含序號時，動詞依下表判斷。**首狀**（不含序號、或序號為「（一）」）使用預設動詞；**二狀（含）以後**的上訴理由狀、答辯狀、準備書狀則使用「補充」或「再提」動詞。

| 書狀類型 | 開頭句格式 |
|------------|------------|
| 起訴狀 | 為就［案由］事件，**起訴**事： |
| 上訴狀 | 為就［案由］事件，**提起上訴**事： |
| 上訴理由狀（首狀：不含序號、或序號為「（一）」） | 為就［案由］事件，**提出上訴理由**事： |
| 上訴理由（二）狀以後（含「（三）」「（四）」等） | 為就［案由］事件，**補充上訴理由**事： |
| 答辯狀（首狀：不含序號、或序號為「（一）」） | 為就［案由］事件，**提出答辯**事： |
| 答辯（二）狀以後（含「（三）」等） | 為就［案由］事件，**再提答辯**事： |
| 準備書狀（首狀：不含序號、或序號為「（一）」） | 為就［案由］事件，**提出準備書**事： |
| 準備書（二）狀以後（含「（三）」等） | 為就［案由］事件，**再提準備書**事： |
| 爭點整理狀 | 為就［案由］事件，**整理爭點**事： |
| 聲請調查證據狀 | 為就［案由］事件，**聲請調查證據**事： |
| 其他書狀 | 依書狀內容自行判斷合適的動詞組合 |

**前置動作 B — 卷宗頁碼檢索（重要）**：在正式生成草稿前，AI 必須主動掃描工作目錄中的既有書狀（特別是同審級之前次書狀），以釐清**既有證據的卷宗頁碼**。**掃描時必須優先尋找同名的 `.md` 檔案**；若無對應之 `.md` 檔，才退而檢索 `.odt` 或 `.docx` 檔。若在舊書狀中查有明確的頁碼，則引證時須帶入；若無或屬新證據，則依規定留空或僅用編號。

**不可遺漏的強制規則**：
1. 結論先行（CRAC）：`一、` 與 `(一)` 標題必須是實質主張的完整句子
2. 裁判引用必須同時提供完整字號 + 要旨原文
3. 起訴狀必須先寫客觀事實，再進入法律論述

**必須等使用者確認草稿後才進入階段 6。**

### 階段 6：保存草稿並自動內部排版

使用者確認草稿後，**連續執行以下步驟，中間不得再向使用者確認**：

1. **檔名生成規則**：保存草稿前，先列出工作目錄中現有的書狀檔案（`.md` 為主），分析其命名規則（例如「案號＋書狀名稱」、「日期＋書狀名稱」、「書狀名稱＋序號」等），依同一規則為本次草稿命名。若工作目錄無既有書狀可供參照，退回預設格式：`書狀名稱_草稿.md`。
2. 依書狀類型分流決策：
   - **一般書狀**（起訴狀、上訴理由狀、答辯狀、準備書狀等）或 **未要求附表格之爭點整理狀** → **直接呼叫內部腳本產出**，執行：`python scripts/build_pleading.py <草稿完整路徑>`，由本技能專屬引擎進行最終 Word 渲染。
   - **爭點整理狀 + 使用者明確要求附表格**（下半部含 JSON）→ **自動呼叫 `pleading-table` 技能**，由其提取下半部 JSON 並生成含爭點表格的最終 DOCX。
3. **自動接續轉檔 ODT**：若使用者下達指令明確要求產出 `.odt` 檔（或將草稿轉成 `.odt` 檔），則在上述 DOCX 生成任務完成後，必須**自動再呼叫 `docx-to-odt` 技能**將該 DOCX 轉為 ODT，無需等待使用者分兩次下達指令。

## 修改補充模式

使用者已有書狀，需修改或補充論點。

### M1：確認修改範圍
- 讀取現有書狀（`.md` 或 `.docx`）
- 確認需求：新增 / 修改 / 刪除 / 調整順序
- **新證據接續編號強制規則**：若使用者提出「新證據」，必須主動搜尋工作區（workspace）內現有的訴訟書狀檔案，檢查目前已經編排到哪個證據編號（例如上證7），然後接續往下編排（延續編為上證8）。

### M2：查詢知識庫（視需要）
若新增或修改的論點需法律依據，依階段 4 查詢，並強制執行 `lawthinker-verify.md` 的深度驗證報告，待確認後才進入 M3。

### M3：修改計畫（需使用者確認）
呈現：現有結構 → 計畫修改 → 修改後預覽。**等使用者確認。**

### M4：更新草稿並內部排版
依確認的計畫輸出更新後的完整 Markdown 草稿，保留所有未涉及的現有內容。
- **強制更新日期**：在輸出更新草稿時，必須嚴格依照【狀首與狀尾日期生成規則】重新生成並覆寫草稿中的狀首與狀尾日期（預設為重新計算與更新為修改當日之後第2日），絕對不可直接照抄舊草稿的過期日期。
- 比照階段 6 呼叫內部專屬腳本 `scripts/build_pleading.py` 完成最終排版。
